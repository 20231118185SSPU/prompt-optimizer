# 转换示例：从粗糙到优秀

> 每个示例展示一个真实场景，原始 Prompt 的问题、应用的规则、优化后的结果。

## 示例 1：模糊的代码请求

**原始 Prompt**：
> 帮我优化一下这个函数

**诊断**：D1=0, D2=0, D3=0, D4=1, D5=0 → 1/10

**应用规则**：R1（精确性注入）+ R3（格式锁定）+ R8（验证门）

**优化后**：
> 重构 `utils/parser.js` 中的 `parseConfig()` 函数。
> - 当前问题：嵌套 5 层 if-else，圈复杂度 > 15
> - 目标：降低圈复杂度到 10 以下，保持函数签名 `(raw: string) => Config` 不变
> - 方法偏好：策略模式或 guard clause
> - 约束：不要改变返回值类型，不要删除现有的错误日志
> - 完成后：给出重构前后的圈复杂度对比

**改动说明**：① 明确了目标文件和函数（R1）② 指定了评估指标和数值目标（R2）③ 加了"完成后对比"的验证要求（R8）

## 示例 7：无验证的实现请求

**原始 Prompt**：
> 帮我写个用户注册接口

**诊断**：D1=1, D2=0, D3=0, D4=0, D5=0 → 1/10

**应用规则**：R1 + R4 + R2 + R8

**优化后**：
> 用 Node.js + Express + PostgreSQL 实现 POST /api/register 接口。
>
> 请求体：`{ email: string, password: string, name: string }`
>
> 功能要求：
> 1. 邮箱格式校验（正则）
> 2. 密码强度校验（>=8 位，含大小写+数字）
> 3. 邮箱唯一性检查
> 4. 密码用 bcrypt 哈希存储
> 5. 成功返回 201 + `{ id, email, name }`
>
> 错误处理：
> - 邮箱已存在 → 409 + `{ error: "邮箱已被注册" }`
> - 校验失败 → 422 + `{ error: "具体错误字段" }`
>
> 约束：使用 Prisma ORM，不引入额外验证库。
>
> 完成后自检：
> 1. 所有错误路径是否都有合适的 HTTP 状态码？
> 2. 密码是否永远不会出现在响应中？
> 3. SQL 注入是否已通过 ORM 防护？
> 4. 并发注册同一邮箱会怎样？

**改动说明**：① 明确了技术栈和接口规范（R1+R4）② 列出了所有边界情况和错误码（R2）③ 加了安全相关的自检清单（R8）

## 示例 8：太长淹没重点

**原始 Prompt**：
> 我最近在做一个项目，是关于电商的，用户反馈说搜索功能不好用，
> 我用的是 Elasticsearch，版本是 8.x，部署在 AWS 上，
> 之前用了 standard analyzer 但是中文分词不好，
> 后来换成了 ik 分词器但是又出了新的问题，
> 现在搜"苹果手机"有时候会搜到水果，
> 我试过加 synonym 但是效果不好，
> 数据量大概 100 万条商品，
> 还有就是搜索结果排序也不太准，
> 价格低的排前面但是用户想要的是相关性高的排前面，
> 你帮我看看怎么优化

**诊断**：D1=1, D2=0, D3=0, D4=1, D5=0 → 2/10

**应用规则**：R6 + R3 + R1 + R4

**优化后**：
> **背景**：电商商品搜索，100 万条数据，Elasticsearch 8.x + IK 分词器，部署在 AWS。
>
> **问题 1（主问题）**：中文搜索歧义
> "苹果手机"搜索结果混入水果类商品。
> 已尝试：standard analyzer（分词不准）→ IK 分词器（解决了分词但出现歧义）→ synonym 映射（效果不佳）。
> 期望：搜索"苹果手机"只返回手机类商品，不返回水果。
>
> **问题 2（次问题）**：排序不准
> 当前默认按价格升序，但用户期望按相关性排序。
> 期望：默认按相关性排序，价格排序作为可选筛选项。
>
> **请按顺序处理**：先解决问题 1，再解决问题 2。
> 每个问题给出：分析原因 → 优化方案 → 具体配置/代码 → 验证方法。

**改动说明**：① 用结构化格式拆分了两个子问题（R6）② 明确了每个问题的现状和期望（R1）③ 指定了处理顺序和输出格式（R3）

## 示例 11：面向 Agent 的项目重构请求

**原始 Prompt**：
> 帮我优化这个项目，让 AI agent 更懂用户

**诊断**：D1=1, D2=0, D3=0, D4=1, D5=0 → 2/10

**应用规则**：R1 + R3 + R4 + R5 + R8 + R9 + R10

**优化后**：
> 你将作为 AI agent 执行一次文档型产品重构任务。
>
> ## 目标
>
> 审计当前 `prompt-optimizer` 项目，将它从“提示词润色工具”升级为“Agent 意图对齐器”。目标是帮助用户把粗糙想法转换成 AI agent 能准确理解、稳定执行、可验证交付的任务契约。
>
> ## 背景
>
> 用户的核心问题不是 prompt 不够漂亮，而是 AI agent 经常误解目标、扩大范围、缺少验收标准。项目应吸收三类思路：五维诊断、澄清访谈、自主思维循环。
>
> ## 范围
>
> 包含：
> 1. 阅读现有 README、方法论、系统 prompt、模板和 skill。
> 2. 重构核心方法论，使其覆盖意图、上下文、范围、交付物、约束、工作流、验收、沉淀。

> 4. 新增面向 agent 的任务简报、澄清访谈、项目上下文模板。
>
> 不包含：
> 1. 不引入 Web 应用或后端服务。
> 2. 不删除旧示例，除非它们与新版方法论冲突。
> 3. 不把用户目标改成纯 prompt 教程。
>
> ## 工作方式
>
> 先阅读项目和参考资料，再说明你发现的结构缺口。随后按最小必要变更修改文档。遇到会改变产品定位的决策，先问一个问题；能从仓库中判断的内容自行判断。
>
> ## 验收标准
>
> 完成后自检：
> - README 是否能让新用户在 1 分钟内理解项目用途？
> - TRANSFORM.md 是否可直接复制给任意 AI 使用？
> - Skill 是否能稳定触发并输出 Agent Brief？
> - 模板是否覆盖直接执行、先澄清、项目上下文沉淀三种关键场景？
> - 是否明确了“不猜关键决策、一次只问一个问题、复杂任务要验收和沉淀”？
>
> ## 完成后的沉淀
>
> 输出本次重构形成的新术语、规则和后续可扩展方向。

**改动说明**：① 把“优化项目”转成明确的文档型重构任务（R1）② 锁定范围和排除项，避免 agent 扩大到写应用（R2+R5）③ 加入工作方式和澄清路由（R9）④ 加入验收和沉淀要求（R8+R10）

## 示例 12：应先澄清的高风险请求

**原始 Prompt**：

>

## 示例 13：XY问题案例

**示例编号**：示例13

**原始 Prompt**：
> 帮我用正则解析这个HTML提取价格

**诊断**：D1=0, D2=0, D3=0, D4=1, D5=0 → 1/10

**识别结果**：解决方案伪装成需求（XY Problem）。用户提出了“用正则解析 HTML”这个具体做法，但真实目标更可能是“稳定提取商品价格”。正则可能在 HTML 结构变化、嵌套标签、转义字符或动态渲染时失效，因此不能直接把用户方案当成已确认需求。

**处理方式**：进入带方案确认的澄清，不直接拒绝用户方案，也不直接输出最终实现任务。

**优化后的第一轮输出**：
> ## 当前理解
>
> 你想从一段 HTML 中提取商品价格。你明确提出的做法是“用正则解析 HTML”，但这可能是解决方案而不是真实需求。如果你的最终目标是稳定提取商品价格，使用 HTML 解析器或结构化选择器通常比正则更可靠。
>
> ## 最大不确定点
>
> 最影响执行路径的是：你是必须按“正则”这个实现方式做，还是只要稳定提取价格即可。
>
> ## 我的问题
>
> 我可以按你说的做【用正则解析 HTML 提取价格】。不过如果你的最终目的是【推测的Y：稳定提取商品价格】，【Z方案：使用 HTML 解析器或 DOM/CSS 选择器提取价格】通常更合适。请确认：A. 就按X做 B. 改用Z做
>
> ## 推荐答案
>
> 推荐选择 B。因为 HTML 不是正则擅长处理的结构化文本，解析器方案更容易处理嵌套标签、属性变化和页面结构调整；如果你有必须使用正则的环境限制，再选择 A。
>
> ## 📌 共识快照 #1
>
> **已确认**：
> - 表面需求：用正则解析 HTML 提取价格
> - 关键决策：尚未确认是坚持正则，还是改用更稳的解析器方案
> - 明确排除：暂未确认
>
> **仍待定**：
> - [假设] 用户的真实目标是稳定提取商品价格，而不是学习正则或满足受限运行环境
