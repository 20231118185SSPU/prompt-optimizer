# 反模式速查手册

> 遇到以下模式时，立即识别并修复。

---

## 反模式 1：模糊型（The Vague Ask）

**识别**：出现"帮我""看看""处理一下""优化优化"

**典型病例**：
```
❌ 帮我看看这段代码
❌ 帮我处理一下这个数据
❌ 帮我优化优化性能
```

**处方**：问自己三个问题——
1. 具体是哪个文件/函数/数据？
2. "看看"是要找 bug？做 review？还是改代码？
3. "好"的标准是什么？

**修复后**：
```
✅ 审查 auth/middleware.go 第 45-80 行，
   重点检查 JWT 过期处理和签名算法安全性，
   用表格列出问题、严重程度和修复建议
```

---

## 反模式 2：长文型（The Wall）

**识别**：超过 300 字的 Prompt，核心问题埋在中间或最后

**典型病例**：
```
❌ 我之前做了个功能，用的是 React，然后遇到了一个问题，
   那个状态管理有点问题，Redux 的 middleware 写得不太好，
   然后我换了 Zustand，但是 Zustand 的持久化好像不支持...
   [500 字后] ...所以你帮我看看那个 middleware 怎么改
```

**处方**：使用"背景-问题-已尝试-期望"结构

**修复后**：
```
✅ 背景：React 应用，从 Redux 迁移到 Zustand
   问题：Zustand 的 persist middleware 在 SSR 场景下报 hydration 错误
   已尝试：加了 skipHydration 选项但状态丢失
   期望：SSR 客户端状态一致，不报 hydration 错误
```

---

## 反模式 3：放养型（The Missing Constraint）

**识别**：没有任何篇幅、数量、范围限制

**典型病例**：
```
❌ 帮我写个周报
❌ 给我讲讲机器学习
❌ 帮我做个方案
```

**处方**：至少加三样东西——篇幅、受众、格式

**修复后**：
```
✅ 写本周工作周报，3-5 个要点，每个 1-2 句。
   发给部门总监，语气正式但不死板。
   结尾加"下周计划"2-3 条。
```

---

## 反模式 4：想当然型（The Assumed Context）

**识别**：你心里清楚但 Prompt 里没写的背景信息

**典型病例**：
```
❌ 这个查询太慢了，帮我优化
   （AI 不知道：什么数据库？多大的表？有没有索引？当前多久？目标多久？）

❌ 这个接口返回不对
   （AI 不知道：什么接口？返回了什么？期望返回什么？在哪里调用的？）
```

**处方**：假装你在跟一个完全不了解项目的新人说话

**修复后**：
```
✅ PostgreSQL 查询优化。
   表 orders 有 800 万行，索引在 (user_id, created_at)。
   当前查询：
   SELECT * FROM orders WHERE user_id = ? AND status = 'pending' ORDER BY created_at DESC
   执行时间 4.5 秒，目标 < 500ms。
   EXPLAIN 显示 Seq Scan。
```

---

## 反模式 5：格式惊喜型（The Format Surprise）

**识别**：你心里有格式预期但没说

**典型病例**：
```
❌ 对比一下这几个方案
   （你想要表格，AI 写了三段散文）

❌ 给我一个计划
   （你想要时间线，AI 给了段落描述）
```

**处方**：直接说"用 ___ 格式输出"

**修复后**：
```
✅ 用表格对比方案 A/B/C，列为：维度、方案A、方案B、方案C、推荐。
   最后一行给结论。
```

---

## 反模式 6：贪多型（The Scope Creep）

**识别**：一个 Prompt 里有 3 个以上不直接关联的任务

**典型病例**：
```
❌ 帮我写个登录页面，加上注册功能，还要忘记密码，
   顺便做个后台管理，对了还有数据库设计
```

**处方**：一次只做一件事。如果确实有关联，用编号 + 优先级

**修复后**：
```
✅ 分步进行：
   第 1 步（本次）：实现登录页面（邮箱 + 密码 + 记住我）
   第 2 步（下次）：注册页面
   第 3 步：忘记密码流程
   请先完成第 1 步，代码放在 src/pages/login.tsx
```

---

## 反模式 7：无验收型（The No-Verification）

**识别**：没有"怎么算做完了"的标准

**典型病例**：
```
❌ 帮我写个单元测试
❌ 帮我部署到 AWS
❌ 帮我写个文档
```

**处方**：加一条"完成后自检：___"

**修复后**：
```
✅ 为 auth 模块写单元测试（Jest）。
   覆盖以下场景：
   1. 正确的邮箱+密码 → 返回 token
   2. 错误密码 → 返回 401
   3. 不存在的邮箱 → 返回 401
   4. 过期 token → 返回 401 并触发刷新逻辑
   
   完成后自检：
   - 覆盖率是否 > 80%？
   - 每个测试是否独立（不依赖执行顺序）？
   - mock 是否正确替换了外部依赖？
```

---

## 反模式 8：照搬型（The Copy-Paste Prompt）

**识别**：直接把一个 System Prompt 或模板不经修改就用

**典型病例**：
```
❌ 直接复制 GPT 的 system prompt 给 Claude
❌ 直接复制别人的 Prompt 不改背景信息
❌ 拿英文 Prompt 直接翻译成中文（某些表达不同）
```

**处方**：理解原始 Prompt 的"意图"而非"文字"，用目标 AI 理解的方式重新表达

**修复后**：
```
原始 Prompt 的意图是："让 AI 扮演资深架构师审查代码"
→ 不要直接复制整段 prompt
→ 而是告诉 AI："请以资深后端架构师的视角审查这段代码，
   重点看安全性和可扩展性，用 [具体格式] 输出"
```

---

## 反模式 9：不澄清型（The Premature Execution）

**识别**：需求会影响目标或架构，但 AI 直接开始写方案或改代码

**典型病例**：
```
❌ 帮我重构整个系统
❌ 帮我做一个更高级的版本
❌ 帮我把产品体验全面升级
```

**处方**：先问一个最高价值问题，并给推荐答案

**修复后**：
```
✅ 当前最影响方案的是重构目标。
   问题：这次重构第一优先级是可维护性、性能、功能扩展还是用户体验？
   推荐：先选可维护性，因为它能降低后续所有改动的成本。
```

---

## 反模式 10：无沉淀型（The No Memory）

**识别**：长期项目里每次都重新解释背景，AI 重复犯同样的错

**典型病例**：
```
❌ 每次都告诉 AI：不要改 public API、不要新增依赖、这个模块叫 workspace 不是 project
```

**处方**：把术语、架构决策、易错点写入项目上下文

**修复后**：
```
✅ 完成任务后输出：
   1. 本次发现的项目术语
   2. 新增或确认的架构规则
   3. Agent 下次必须避免的错误
   4. 建议写入 PROJECT-CONTEXT.md 的内容
```

---

## 反模式 11：提前完成型（The Premature Completion）

**识别**：AI 没有跑测试、没检查边界、没说明验证方式，就宣布“已完成”

**典型病例**：
```
❌ 已完成优化。
❌ 代码已经修好了。
❌ 这个方案没问题。
```

**处方**：在 Prompt 中加入明确验收门

**修复后**：
```
✅ 完成后必须说明：
   - 实际修改了哪些文件
   - 如何验证
   - 哪些验收项已通过
   - 哪些风险仍然存在
```

## 快速识别口诀

| 你说了... | 反模式 | 你需要加... |
|----------|--------|-----------|
| "帮我看看" | 模糊型 | 具体看哪里、看什么 |
| "写个 ___" | 放养型 | 篇幅、受众、格式 |
| [超长描述] | 长文型 | 结构化拆分 |
| "优化/修复" | 想当然型 | 技术栈、错误信息、期望行为 |
| "对比/比较" | 格式惊喜型 | 表格还是列表 |
| "A + B + C" | 贪多型 | 只留 A，B 和 C 下次再说 |
| "做个 ___" | 无验收型 | 完成后自检清单 |
| [复制粘贴] | 照搬型 | 理解意图，重新表达 |
| "越强越好" | 不澄清型 | 最高价值问题 |
| 反复解释规则 | 无沉淀型 | 项目上下文 |
| "已完成"但没验证 | 提前完成型 | 验收门 |
