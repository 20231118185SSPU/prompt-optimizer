# 使用说明

> 使用本项目的核心动作：先把想法变成 Agent Brief，再让 AI agent 执行。

## 1. 最短用法

安装后只需要这样说：

```text
$optimize-prompt 优化：帮我做一个用户登录功能
```

Claude Code：

```text
/optimize-prompt 优化：帮我做一个用户登录功能
```

如果你的工具支持根据描述自动调用 skill，也可以直接说：

```text
优化：帮我做一个用户登录功能
```

AI 会输出一个可交给 agent 执行的 Agent Brief。

把你的原始想法发给已接入本项目的 AI：

```text
帮我做一个用户登录功能
```

AI 应该先输出类似结构：

```markdown
## 优化后的 Prompt

你将作为 AI agent 执行以下任务。

### 目标
...

### 范围
...

### 工作方式
...

### 验收标准
...
```

然后你把这份优化后的 Prompt 交给真正执行任务的 agent。

## 2. 可选模式

多数时候不用写模式。需要更精细控制时再用下面这些前缀。

### `[直出]`

只要最终 prompt，不要诊断。

```text
[直出] 帮我写一个适合 Codex 执行的重构任务
```

### `[访谈]`

你还没想清楚时使用。AI 会一次只问一个问题。

```text
[访谈] 我想把系统重新设计一下，越强越好
```

适合：

- 目标很大
- 方向不确定
- 可能影响架构、成本、权限或数据

### `[Agent Brief]`

直接输出完整任务简报。

```text
[Agent Brief] 帮我把一个粗糙产品想法整理成可以交给 Claude Code 实现的任务
```

适合：

- 编程任务
- 重构任务
- 调研任务
- 多步骤复杂任务

### `[项目上下文]`

生成可保存的项目记忆。

```text
[项目上下文] 根据下面这段项目介绍，生成给 AI agent 每次任务前读取的上下文文件
```

适合：

- 长期项目
- 团队协作
- agent 经常重复误解术语或架构规则

## 3. 推荐工作流

### 简单任务

```text
[直出] 把下面这句话改成更明确的 AI 指令：...
```

### 复杂任务

```text
[Agent Brief] 我想让 AI agent 帮我实现/重构/分析...
```

拿到 Agent Brief 后，先检查：

- 目标是否符合你的真实意图
- 范围是否过大或过小
- 禁止事项是否足够
- 验收标准是否能判断完成

确认后再交给执行 agent。

### 高风险任务

```text
[访谈] 帮我重构整个项目
```

先让 AI 问清方向，不要直接让 agent 动手。

## 4. 编程任务示例

原始请求：

```text
帮我修一下登录 bug
```

更好的输入：

```text
[Agent Brief]
我想让 AI agent 修复登录 bug。
目前只知道登录接口返回 500，但还没定位根因。
请帮我把这个请求整理成一个适合 coding agent 执行的任务。
```

期望输出会要求 agent：

- 先读取相关路由、服务和错误日志
- 复现问题
- 定位根因
- 做最小修复
- 增加或更新测试
- 说明验证结果

## 5. 调研任务示例

```text
[Agent Brief]
帮我调研 AI 编程工具，但不要泛泛介绍。
目标是给 8 人 TypeScript 团队做选型。
```

期望输出会补齐：

- 时间范围
- 信息来源偏好
- 评估维度
- 输出格式
- 事实和观点区分
- 推荐标准

## 6. 项目沉淀用法

每次复杂任务结束后，让 agent 输出：

```text
请把本次任务中值得沉淀的内容整理成 PROJECT-CONTEXT.md 更新建议：
1. 新术语
2. 新规则
3. 易错点
4. 下次任务应默认读取的上下文
```

这一步能减少下一次 agent 重复犯错。

## 7. 判断输出好坏

好的优化结果应该满足：

- agent 不需要猜主目标
- 明确包含“不做什么”
- 有具体交付物
- 有工作方式
- 有验收标准
- 信息不足时知道先问什么
- 复杂项目有沉淀要求

不好的优化结果通常只是：

- 更礼貌
- 更长
- 更多形容词
- 没有验收标准
- 没有范围边界
- 没有处理信息不足
