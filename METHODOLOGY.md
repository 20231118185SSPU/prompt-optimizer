# Agent 意图对齐方法论

> 目标：让 AI agent 精准理解用户真正想要什么，并用可验证、可迭代、可沉淀的方式执行。

---

## 1. 核心判断

Prompt 优化的重点不是“写得更漂亮”，而是降低三类偏差：

1. **意图偏差**：用户想要 A，agent 理解成 B。
2. **范围偏差**：用户只想改一处，agent 扩大到整套方案。
3. **验收偏差**：agent 以为完成了，用户发现不符合实际目标。

因此，一个高质量 prompt 应该像一份轻量任务契约，而不是一句请求。

---

## 2. 五维诊断模型

每个 Prompt 都可以在 5 个维度上打分，每个维度 0-2 分：

| 维度 | 评什么 | 缺了会怎样 |
| --- | --- | --- |
| **D1 精确性** | 目标是否明确、动词是否精确、对象是否具体 | AI 猜测意图，输出笼统 |
| **D2 约束性** | 范围、边界、数量、排除项、权限 | AI 跑偏、加戏、扩大修改 |
| **D3 结构性** | 输出格式、阶段顺序、交付物形态 | 格式不对，信息散乱 |
| **D4 上下文** | 背景、受众、用途、项目状态、行为风格 | 深度不对，理解偏差 |
| **D5 验证性** | 成功标准、测试、自检、例子 | 看起来完成，实际不可用 |

评分标准：

- **0 分（缺失）**：完全没有涉及。
- **1 分（隐含）**：有暗示但不明确，AI 需要猜。
- **2 分（明确）**：直接、清晰、无歧义。

总分解释：

- **0-5 分**：必须先优化或澄清。
- **6-7 分**：可执行，但存在偏差风险。
- **8-10 分**：可执行，可做轻量增强。

---

## 3. Agent Brief 八组件

面向 agent 的最终输出，应优先组装为以下结构：

| 组件 | 作用 | 问题 |
| --- | --- | --- |
| **INTENT** | 用户真正想达成什么 | 为什么要做这件事？ |
| **CONTEXT** | 背景和已有状态 | agent 需要先知道什么？ |
| **TARGET** | 具体任务对象 | 改什么、写什么、分析什么？ |
| **SCOPE** | 范围和边界 | 包括什么，不包括什么？ |
| **DELIVERABLES** | 交付物 | 最终要输出什么形态？ |
| **CONSTRAINTS** | 技术、风格、权限、反面约束 | 不允许做什么？必须保持什么？ |
| **WORKFLOW** | 执行策略 | 直接执行、先问、先设计、分阶段？ |
| **ACCEPTANCE** | 验收与自检 | 怎么判断做对了？ |

如果任务涉及代码库、长期项目或多轮协作，再追加：

- **MEMORY**：需要沉淀到项目规则、上下文或经验库的内容。
- **RISKS**：最容易误解、最容易做坏的点。

---

## 4. 澄清优先级

不是所有缺失信息都值得追问。按以下顺序判断：

1. **会改变目标的缺失**：必须问。
2. **会改变架构或交付物的缺失**：必须问。
3. **会影响安全、数据、成本、权限的缺失**：必须问。
4. **只影响表达风格或细节的缺失**：可以给默认假设并继续。
5. **可通过阅读代码、文档、网页获得的信息**：agent 应自己探索，不问用户。

澄清规则：

- 一次只问一个问题。
- 每个问题都给出推荐答案。
- 问题必须能减少执行偏差。
- 不问“你还有什么补充吗”这种低价值问题。

---

## 5. 自主思维循环

复杂任务应要求 agent 使用六阶段循环：

1. **感知**：阅读需求、上下文、相关文件，区分用户原话和真实目标。
2. **自设计**：判断需要什么角色、能力和策略。
3. **深度思考**：比较方案、识别风险、确定最小可行路径。
4. **执行**：小步实现，每步可验证。
5. **反思**：从技术、业务、架构、体验四个维度自检。
6. **沉淀**：记录项目模式、踩坑、规则建议和后续可复用模板。

这不是要求 agent 展示冗长推理，而是要求它采用稳定的工作过程，并把关键决策和验收结果显性化。

---

## 6. 转换规则

按优先级应用以下规则：

### R1 精确性注入

触发：出现“帮我”“弄一下”“优化一下”“看看”等模糊动词。

操作：替换为“精确动词 + 具体对象 + 目标结果”。

示例：

- 原始：帮我优化这个项目
- 优化：审计当前项目的文档结构和 prompt 转换流程，找出导致 agent 误解用户意图的缺口，并重构核心方法论和模板。

### R2 边界锁定

触发：没有范围、权限、数量、排除项。

操作：补充“只做什么 / 不做什么 / 需要确认后才能做什么”。

### R3 交付物锁定

触发：没有指定最终输出形态。

操作：指定格式，如 Agent Brief、PRD、代码补丁、审查报告、决策矩阵、JSON。

### R4 上下文锚定

触发：没有说明背景、受众、用途、项目状态。

操作：补充最小必要背景，包括用户是谁、为什么需要、输出将被谁使用。

### R5 反面约束

触发：agent 自由度过大。

操作：添加 1-3 条“不要做”。优先限制最容易造成损失的行为。

### R6 范围裁剪

触发：一个 prompt 里有 3 个以上不相关任务。

操作：拆成主任务、次任务、后续任务。当前只让 agent 处理一个主目标。

### R7 示例嵌入

触发：主观任务、风格任务、复杂格式任务没有例子。

操作：给 1 个正例，必要时给 1 个反例。

### R8 验证门

触发：没有成功标准或测试策略。

操作：添加可检查的验收项、自检项、测试命令或人工验收条件。

### R9 意图路由

触发：用户输入可能对应不同交互模式。

操作：加入路由：

- 信息足够且风险低：直接执行。
- 信息不足且影响目标：先问一个关键问题。
- 任务复杂或高风险：先输出方案和验收标准，确认后执行。
- 只读分析：不要修改文件。
- 代码修改：先阅读相关上下文，再最小变更。

### R10 沉淀门

触发：长期项目、代码库、团队规则、反复任务。

操作：要求 agent 在完成后输出“应沉淀的项目规则 / 术语 / 风险 / 模板”。

---

## 7. 输出模式

### 模式 A：直接优化

适合简单 prompt。

输出：

1. 优化后的 Prompt
2. 诊断表
3. 改动记录

### 模式 B：澄清访谈

适合用户还没想清楚、需求会影响方向的任务。

输出：

1. 当前理解
2. 最大不确定点
3. 一个问题
4. 推荐答案

### 模式 C：Agent Brief

适合交给 AI agent 执行的开发、研究、重构、写作、分析任务。

输出：

1. 任务简报
2. 执行策略
3. 验收标准
4. 反面约束
5. 沉淀要求

### 模式 D：项目上下文沉淀

适合长期项目。

输出：

1. 项目术语
2. 架构/流程约定
3. 已知风险
4. Agent 工作规则
5. 需要定期更新的内容

---

## 8. 反模式

### 1. 只润色，不对齐

把“帮我做个功能”改成“请实现一个功能”，仍然没有解决目标、边界、验收缺失。

### 2. 替用户做产品决策

优化 prompt 可以补全结构，但不能擅自改变用户目标。

### 3. 一次问太多

多个问题会让用户难以回答，也会隐藏最关键的决策点。

### 4. 默认直接执行

高风险任务必须先对齐方案和验收。

### 5. 过度规则化

prompt 太长、约束太多，会让 agent 执行迟钝。只保留能改变行为的规则。

### 6. 缺少验证门

没有验收标准的任务，agent 很容易提前宣布完成。

### 7. 缺少上下文记忆

每次都重新解释项目，会让 agent 重复犯错。长期项目必须沉淀术语、架构和决策。

---

## 9. 最终自检清单

发送给 agent 前，检查：

- [ ] agent 是否知道真正目标，而不只是表面任务？
- [ ] 是否说明了必须读取或参考的上下文？
- [ ] 是否明确了范围、边界和禁止事项？
- [ ] 是否定义了交付物格式？
- [ ] 是否设置了信息不足时的行为？
- [ ] 是否有可检查的验收标准？
- [ ] 是否要求复杂任务完成后反思和沉淀？

如果 3 项以上无法打勾，先优化或进入澄清访谈。
