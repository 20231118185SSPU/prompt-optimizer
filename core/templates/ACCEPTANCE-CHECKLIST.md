# 验收清单库

> 分类型可复制验收清单库。用于在 D5=0 或缺少明确验收标准时，强制补充可执行的验收标准。

## 使用规则

- 触发：D5=0 或无明确验收标准。
- 执行：按任务类型选择清单，写入 Agent Brief 的 `验收标准` 或 `完成后自检`。
- 禁止：不得使用抽象验收标准（"要好用""要稳定"）。
- 禁止：不得省略验收标准（即使用户未提及）。
- 禁止：不得使用不可执行的标准（"符合最佳实践"）。

## 代码类

```markdown
完成后自检：
- [ ] 执行测试命令：`npm test` 是否全部通过？
- [ ] 类型检查：`tsc --noEmit` 是否无错误？
- [ ] 代码质量：圈复杂度是否<10？（工具：`eslint`）
- [ ] 是否引入新依赖？如是，说明理由
- [ ] 是否修改 public API？如是，标记 breaking change
```

## 调研类

```markdown
完成后自检：
- [ ] 是否覆盖所有要求的对比维度？
- [ ] 每个结论是否有数据来源？
- [ ] 是否给出明确推荐（非"各有优劣"）？
- [ ] 推荐理由是否与用户场景匹配？
```

## 写作类

```markdown
完成后自检：
- [ ] 篇幅：实际___字，要求___字
- [ ] 语气：是否符合受众（技术/商务/学术）？
- [ ] 结构：是否包含要求的所有章节？
- [ ] 术语：是否准确无歧义？
```

## 重构类

```markdown
完成后自检：
- [ ] 现有测试是否全部通过？
- [ ] 对外接口、数据格式或用户可见行为是否保持兼容？
- [ ] 是否删除或减少了指定重复、复杂度或耦合问题？
- [ ] 是否说明了修改范围、未修改范围和潜在风险？
- [ ] 如存在性能或复杂度目标，是否给出命令、数字或检查结果？
```

## 部署类

```markdown
完成后自检：
- [ ] 部署前检查命令是否通过？
- [ ] 配置、密钥、环境变量是否按目标环境确认？
- [ ] 是否提供回滚步骤和触发回滚的明确条件？
- [ ] 是否确认监控、日志或健康检查可用？
- [ ] 是否记录部署版本、影响范围和验证时间点？
```
