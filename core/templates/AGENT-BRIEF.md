# Agent Brief 模板

> 用于把一个想法转换成可交给 AI agent 执行的任务契约。

```markdown
你将作为 AI agent 执行以下任务。

## 目标

[用 1-3 句话说明用户真正想达成什么。不要只复述表面请求。]

## 信息来源与置信度

- [原文]：[用户明确说过的目标、对象或约束]
- [推断]：[从上下文/惯例合理推断的信息，用户应扫一眼确认]
- [假设]：[无依据的默认值；执行时必须先验证，验证失败则先提问]

## 背景

- 用户场景：[为什么需要这个任务]
- 当前状态：[已有材料、项目状态、已尝试内容]
- 目标受众：[最终结果给谁用]

## 任务对象

- 主要对象：[文件 / 项目 / 主题 / 产品 / 文档 / 代码模块]
- 相关对象：[需要参考但不一定修改的内容]

## 范围

包含：

1. [必须完成的范围]
2. [必须完成的范围]

不包含：

1. [明确排除项]
2. [需要用户确认后才能做的事]

## 工作方式

1. 先感知：阅读相关上下文，复述你对目标和边界的理解。
2. 再自设计：说明你需要采用的角色、能力和执行策略。
3. 深度思考：列出关键方案、风险和取舍。
4. 执行：小步完成，每步可验证。
5. 反思：完成后从目标、范围、质量、风险四个维度自检。
6. 沉淀：输出可复用的规则、术语、经验或后续建议。

如果发现关键信息缺失，并且会改变目标或交付物，先只问一个问题，并给出你的推荐答案。

如果存在 `[假设]`，执行时先验证该假设；验证失败或会改变目标、范围、交付物、权限或验收标准时，先只问一个问题，并给出推荐答案。

## 执行策略

- 信息足够且低风险 → 直接执行。
- 关键信息缺失且会改变目标 → 停下来，只问一个问题，附推荐答案。
- 信息可从文件/文档/日志/URL获得 → 自己读取，不问用户。
- 高风险操作（删除/生产/不可逆）→ 先输出方案和影响面，等待确认。
- 只读分析任务 → 禁止修改任何文件。
- 代码修改任务 → 先读上下文，再最小变更。

## 交付物

- [交付物 1：格式、篇幅、结构]
- [交付物 2：如适用]

## 约束

- 不要：[最重要的禁止行为]
- 不要：[第二个禁止行为]
- 禁止用"尽量/建议/一般来说"替代明确的权限、范围或验收标准。
- 必须保持：[接口 / 风格 / 既有规则 / 兼容性]
- 优先使用：[已有材料 / 项目现有模式 / 官方来源]

## 验收标准

从 `templates/ACCEPTANCE-CHECKLIST.md` 按任务类型选择或改写为可执行验收项。不得只写“要好用”“要稳定”“符合最佳实践”等抽象标准。

完成后自检：

- [ ] 是否完整满足目标？
- [ ] 是否没有超出范围？
- [ ] 是否遵守所有约束？
- [ ] 是否给出了可验证的结果？
- [ ] 如果修改了项目，是否说明改了什么、为什么改、如何验证？

## 契约回验

输出前必须自答回验四问，失败项必须先修正再交付：

- [ ] Q1 意图保真：最终结果解决的还是用户原始问题。
- [ ] Q2 无擅自决策：没有替用户做产品/技术方向决策。
- [ ] Q3 可独立执行：一个新 agent 拿到契约无需猜测即可执行。
- [ ] Q4 验收可判定：每条验收标准都能用命令、数字或清单检查。

## 输出格式

[指定最终输出结构，例如：先结论，后过程；表格；PRD；代码补丁摘要；Markdown 文档等。]
```
