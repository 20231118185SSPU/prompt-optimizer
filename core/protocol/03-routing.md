# 智能路由

> 协议内核 SSOT 文件。内容由 `build/` 脚本生成到 `dist/`，禁止手改 `dist/`。

## 7. 智能路由

用户不应该记忆模式名。优化器应先自动判断任务类型，再选择输出形态。

## 阶段2：智能路由（v3.0 重构版）

**当前问题**：路由规则模糊，AI容易误判。

**解决方案**：v3.0 静默三档路由 + v2.0 三层决策树。

v3.0 的核心设计原则：**对齐的存在感与任务风险成正比，与任务频率成反比。**

- 简单高频指令：零感知，直接执行。
- 中等复杂指令：静默补全，微披露，不等待。
- 高风险/目标不明指令：停下澄清，一次一问。

---

## v3.0 静默三档路由

每条用户指令 → 3 秒内完成内部评估（不输出评估过程）→ 三档分流。

三档路由是 v2.0 三层决策树的外层包装：先决定"静默度"（用户感知多少），再由 v2.0 三层决定"输出形态"（输出什么格式）。

```text
每条用户指令 → 3 秒内完成内部评估（不输出评估过程）→ 三档分流

┌─ 档位 A：直通（Pass-through）—— 预计覆盖 ~60% 指令
│   条件：单步简单 + 无高风险信号 + 意图明确（五维快评 ≥8 分）
│   行为：完全不提"优化"二字，直接执行。
│         内部仍做一件事：静默套用 R8 验证门（执行完自验证再交付）。
│   用户感知：零。没有任何额外输出、没有任何等待。
│
├─ 档位 B：静默对齐(Silent Align)—— 预计覆盖 ~30% 指令
│   条件：有缺口但可从项目上下文补全（缺口 ∈ .align/ 可回答范围）；或五维 6-7 分且 0 分项可补全
│   行为：agent 在内部完成八组件补全（含可选 MEMORY/RISKS） + 转换规则应用，然后【直接开始执行】。
│         仅在回复开头用 1-3 行披露关键对齐假设：
│         「按项目规范：本次改动限于 auth 模块，不动数据库 schema，
│           完成后跑 test:auth。若与你预期不符请打断我。」
│         然后立刻继续干活，不等待确认。
│   用户感知：一段可扫读的对齐披露，无停顿。
│
└─ 档位 C：浮出澄清（Surface）—— 预计覆盖 ~10% 指令
    条件（沿用 v2.0 硬性门槛，一条不弱化）：
      · 高风险信号（生产/删除/数据库/目标不明的重构）且缺目标/范围/约束/验收
      · 诊断总分 <6
      · [假设] >2 条
    行为：停下，一次只问一个问题 + 给推荐答案（v2.0 澄清协议原样保留）。
    用户感知：被打断——但这正是该被打断的 10%。
```

### 档位 A：直通（Pass-through）

| 属性 | 定义 |
| --- | --- |
| **条件** | 单步简单 + 无高风险信号 + 意图明确（五维快评 ≥8 分） |
| **行为** | 完全不提"优化"二字，直接执行。内部静默套用 R8 验证门（执行完自验证再交付）。 |
| **用户感知** | 零。没有任何额外输出、没有任何等待。 |
| **失败/升档条件** | 执行中发现高风险信号或目标偏离 → 升档 B 或 C；R8 验证失败 → 修复后再交付。 |

### 档位 B：静默对齐（Silent Align）

| 属性 | 定义 |
| --- | --- |
| **条件** | 有缺口但可从项目上下文补全（缺口 ∈ `.align/` 可回答范围）；或五维快评 6-7 分且 0 分项可从 .align/ 补全 |
| **行为** | agent 在内部完成八组件补全（含可选 MEMORY/RISKS） + 转换规则应用，然后直接开始执行。仅在回复开头用 1-3 行披露关键对齐假设，然后立刻继续干活，不等待确认。 |
| **用户感知** | 一段可扫读的对齐披露，无停顿。 |
| **失败/升档条件** | 披露超过 5 行 → 升档 C；发现高风险信号且 `.align/` 无法补全 → 升档 C。 |

### 档位 C：浮出澄清（Surface）

| 属性 | 定义 |
| --- | --- |
| **条件** | 高风险信号且缺目标/范围/约束/验收；或诊断总分 <6；或 `[假设]` >2 条 |
| **行为** | 停下，一次只问一个问题 + 给推荐答案（v2.0 澄清协议原样保留）。 |
| **用户感知** | 被打断——但这正是该被打断的 10%。 |
| **失败/升档条件** | 无降档。澄清完成后根据结果进入 A 或 B。 |

## 显式前缀覆盖规则

显式前缀是对三档路由的强制覆盖，优先级最高。

| 前缀 | 覆盖行为 | 说明 |
| --- | --- | --- |
| `[直出]` | 直接输出优化后的 Prompt，不输出诊断、路由日志、改动记录 | v2.0 Direct Output 行为保留 |
| `[访谈]` | 强制进入澄清访谈，一次一问 + 推荐答案 | v2.0 Clarify First 行为保留 |
| `[Agent Brief]` | 输出完整 Agent Brief 文档 | v2.0 Agent Brief 行为保留 |
| `[项目上下文]` | 输出项目上下文/记忆文件草案 | v2.0 Project Context 行为保留 |
| `优化：` | 输出完整 Agent Brief 文档（v2.0 行为） | 满足"我就想看优化结果"的场景 |

显式前缀不跳过安全阀：即使用户使用 `[直出]`，如果命中高风险信号且缺少目标/范围/约束/验收，仍必须先探查或澄清。

## 防退化红线

以下红线不可协商，是重做条件而非注意事项：

1. **档位 A 的"直通"不等于放弃验证——R8 验证门在所有档位强制生效。** 档位 A 直通时仍必须静默执行 R8 验证门（执行完自验证再交付）。
2. **档位 B 的披露不得超过 5 行，超过说明该走档位 C。** 披露超 5 行 = 升档 C，不是"多写几行也没关系"。
3. **禁止把档位 C 的场景静默处理：高风险静默假设 = 输出无效，必须重做。** 继承 v2.0 Quality Bar，高风险场景静默假设等同于输出无效。

## v2.0 三层智能路由（三档路由内部逻辑）

三档路由内部使用 v2.0 三层决策树判断任务类型和输出形态。安全阀第一层原样保留且优先级最高。

### 第一层：安全阀判断（强制优先级）

```text
用户指令

  │

  ├─ 包含高风险信号？

  │  ├─ 是，且缺少目标、范围、约束或验收 → 强制进入【意图探查】

  │  │  信号：生产环境/数据库/删除/重构（目标或范围不明时）/越X越好/提升性能（指标不明时）

  │  ├─ 是，但目标、范围、约束和验收已明确 → 继续路由，并在 Agent Brief 中加入影响面分析和风险确认

  │  └─ 否 → 继续

  │

  ├─ 诊断总分<6？

  │  ├─ 是 → 强制进入【澄清访谈】

  │  └─ 否 → 继续

  │

  └─ D5验证性=0？

     ├─ 是 → 强制补全验收标准

     └─ 否 → 继续

```

### 第二层：任务类型判断

```text
通过安全阀

  │

  ├─ 编程类任务？

  │  关键词：实现/重构/调试/部署/测试

  │  → 【Agent Brief - 代码模板】

  │

  ├─ 调研类任务？

  │  关键词：对比/分析/调研/评估/选型

  │  → 【Agent Brief - 调研模板】

  │

  ├─ 写作类任务？

  │  关键词：写/撰写/文档/PRD/报告

  │  → 【Agent Brief - 写作模板】

  │

  ├─ 项目沉淀？

  │  关键词：记住/下次/以后/规则/团队约定

  │  → 【项目上下文模板】

  │

  └─ 其他

     → 【通用Agent Brief】

```

### 第三层：复杂度判断

```text
确定任务类型

  │

  ├─ 单步简单任务？

  │  特征：改一个文件/单一动作/无依赖

  │  → 【基础优化模式】

  │

  ├─ 多步复杂任务？

  │  特征：3+文件/跨模块/需要架构设计

  │  → 【Agent Brief + 分阶段审批门】

  │

  └─ 长期迭代任务？

     特征：会有后续版本/需要上下文沉淀

     → 【Agent Brief + 项目上下文沉淀】

```

## 输出模式映射

v3.0 输出模式 = 档位（静默度） × v2.0 模式（输出形态）。

| v2.0 模式 | 触发条件 | 输出 | v3 默认档位 |
| --- | --- | --- | --- |
| Basic Optimize / 基础优化 | 通过安全阀，且第三层判定为单步简单任务 | 优化后的 Prompt + 简短诊断 + 路由决策日志 + 改动记录 | 档位 A（直通） |
| Agent Brief | 开发、重构、调研、设计、调试、执行类多步骤任务 | 完整任务简报，包含目标、背景、范围、交付物、工作方式、约束、验收标准和沉淀要求 | 档位 B（静默对齐） |
| Clarify First / 先澄清 | 安全阀命中低分、关键缺失或 `[假设]` 超过2条 | 当前理解、最大不确定点、一个问题、推荐答案、共识快照 | 档位 C（浮出澄清） |
| Project Context / 项目上下文 | 用户想沉淀规则、术语、团队约定、项目记忆 | 可保存的项目上下文 / 记忆文件草案 | 档位 B（静默对齐） |
| Direct Output / 直出 | 用户明确 `[直出]`、只给结果，且未命中安全阀 | 只输出优化后的 Prompt | 档位 A（直通） |

档位与 v2.0 模式的对应关系是默认值，可被显式前缀覆盖。

## 路由日志

### v3.0 静默模式下的路由日志

档位 A 和档位 B 默认不输出路由决策日志（静默模式）。档位 C 和显式前缀模式仍输出路由日志。

### v2.0 路由日志（档位 C 和显式模式必须输出）

除非用户使用 `[直出]` 且未命中安全阀，否则档位 C 和显式前缀模式必须输出路由决策日志：

```markdown
## 路由决策日志

**v3 档位** → 档位 C（浮出澄清）
- 升档原因：高风险信号 + 缺少目标/范围/约束/验收

**第一层：安全阀** ⚠️ 命中
- 高风险信号：检测到（生产环境/数据库）
- 诊断总分：3/10（<6）
- 验证性：0/2（未明确）

**第二层：任务类型** → 编程类任务
- 检测关键词："重构"
- 匹配模板：Agent Brief - 代码模板

**第三层：复杂度** → 多步复杂任务
- 影响范围：跨模块
- 依赖关系：有跨模块依赖
- 最终模式：Clarify First

**执行路径**：停下 → 一次一问 + 推荐答案 → 用户确认后进入 Agent Brief
```

## 路由原则

- 不让用户选择模式，除非用户主动指定。
- 简单任务走基础优化（档位 A），不输出过重结构。
- 多步骤 agent 任务走 Agent Brief（档位 B），开头披露关键假设后直接执行。
- 高风险不确定任务先澄清（档位 C），一次只问一个问题。
- 长期项目和重复协作任务自动加入沉淀门。
- 能通过读取代码、文档、日志、网页获得的信息，不问用户，写入未来 agent 的感知步骤。
- `.align/` 存在时，档位 B 的缺口补全优先从 `.align/` 取材（lessons → spec → context 读取顺序）。
