# 自动沉淀协议

> 协议内核 SSOT 文件。内容由 `build/` 脚本生成到 `dist/`，禁止手改 `dist/`。

## 定位

Prompt Optimizer v3.0 是一个可注入任意项目的对齐运行时（Alignment Runtime）：接入时为项目生成开发规范，运行时让每条开发指令在执行前静默通过意图对齐管线，执行后把经验沉淀回项目。

自动沉淀不是写总结，也不是记录"本次任务很顺利"。每条沉淀必须能改变下次 agent 的行为，否则删掉。

---

## 沉淀目标

执行 `align-init` 后，目标项目获得 `.align/` 运行时目录：

```text
.align/
├── spec.md            # 项目开发规范
├── context.md         # 项目上下文契约
├── lessons.md         # 经验规则（沉淀门自动追加，agent 每次任务前必读）
├── decisions.log.md   # 重大决策日志（档位 C 澄清的结论自动归档）
├── check-commands.txt # R8 验证门命令清单（align-check.sh 逐行执行）
├── debt.md            # 债务台账（align-check.sh 扫描 TODO/FIXME/HACK 自动登记）
├── route.conf         # 路由配置（白名单解析，绝不 source）
├── align-route.sh     # 确定性信号评分路由器（UserPromptSubmit hook）
└── align-check.sh     # 一键验证脚本（跑 check-commands.txt + 债务扫描）
```

### 读取顺序

agent 的读取顺序写死在挂载区，固定为：

1. `.align/lessons.md`（最易违反的最先读）
2. `.align/spec.md`（项目规范）
3. `.align/context.md`（项目上下文）

最易违反、最新形成的经验必须优先进入 agent 视野。

### 大小纪律

四个文件都有大小纪律，防止膨胀成没人读的文档：

| 文件 | 最大行数 | 超限处理 |
| --- | --- | --- |
| `spec.md` | ≤150 行 | 精简或拆分到 `context.md` |
| `context.md` | ≤100 行 | 精简或归档旧决策到 `decisions.log.md` |
| `lessons.md` | 每条 ≤2 行，超 50 条归档 | 旧条目归档到 `.align/archive/lessons-{date}.md` |
| `decisions.log.md` | 每条 ≤5 行 | 旧决策归档到 `.align/archive/decisions-{date}.md` |

## 触发条件

满足以下任一条件时，任务结束必须考虑沉淀（即门 5 沉淀门触发）：

### 1. 踩坑

本次执行中出现可复用的错误模式或规避规则。

**判定标准**：如果下次遇到类似场景，这条经验能避免重复犯错。

**示例信号**：agent 执行中报错、测试失败、构建中断、用户指出"你又搞错了"。

### 2. 用户纠正

用户指出 agent 的理解、范围、验收或执行方式有偏差。

**判定标准**：用户的纠正改变了对齐结果，且下次类似任务可能再次偏差。

**示例信号**：用户说"不是这个意思"、"范围太大了"、"不要动这个文件"、"验收不是这样的"。

### 3. 新约定

任务中形成新的项目规则、术语、边界或质量门槛。

**判定标准**：新约定是任务过程中涌现的，不是 `spec.md` 中已有的。

**示例信号**：用户说"以后都按这个来"、"记住这个规则"、"团队约定是"。

### 4. 推翻假设

原有 `[假设]` 被证明错误，且下次可能再次影响执行。

**判定标准**：假设被证伪，且该假设的默认值可能在下次任务中再次被采用。

**示例信号**：agent 执行时发现 `[假设] 允许修改内部实现` 实际上不允许，或 `[假设] 使用 PostgreSQL` 实际是 MySQL。

### 5. 高风险决策确认

档位 C 澄清的结论涉及高风险操作（生产/数据库/删除/不可逆），用户已确认方向。

**判定标准**：决策影响后续方向、架构、权限或高风险边界。

**示例信号**：用户确认"就按方案 A 做"、"允许修改数据库 schema"、"可以删除旧代码"。

### 6. 可复用模板

出现可复用的 Agent Brief、检查清单或决策模式。

**判定标准**：本次任务形成的结构可被未来类似任务直接套用。

**示例信号**：用户说"这个格式不错，以后都这样"、agent 发现某检查清单可通用。

## 不该沉淀的内容

以下内容不得写入 `.align/`，违反 = 沉淀无效，应删除：

### 1. 空洞总结

"本次任务很顺利"、"代码已优化"、"测试通过"等不改变下次 agent 行为的记录。

**判定标准**：删掉这条，下次 agent 的行为会不同吗？如果不会，不该沉淀。

### 2. 一次性任务信息

只对当前任务有效、不可复用的信息。

**判定标准**：下次任何任务都不会再遇到这个场景。

**示例**："本次修改了 `src/auth/login.js` 第 42 行"（具体行号无复用价值）。

### 3. 不能改变行为的记录

无法转化为"下次执行规则"的信息。

**判定标准**：无法写成"遇到 X 场景 → 执行 Y 规则"的格式。

**示例**："用户今天心情不太好"（无法转化为执行规则）。

### 4. 未确认的推断

不能把 `[推断]` 或 `[假设]` 直接写成项目事实。

**判定标准**：沉淀内容必须有用户确认或可验证的证据。

**示例**：agent 推断"项目使用 ESLint"但未验证 → 不该沉淀为事实，应先验证或标注为待确认。

### 5. 与 spec.md 重复的内容

`spec.md` 已有的规则不需要在 `lessons.md` 中重复。

**判定标准**：检查 `spec.md` 是否已包含相同规则。

## 写入格式

### lessons.md 格式

每条不超过 2 行，格式：

```text
- [场景/触发条件] 规则：xxx → 下次执行：xxx
```

示例：

```text
- [数据库变更] 规则：schema 变更必须先在 staging 验证 → 下次执行：涉及 migration 文件时先跑 `npm run db:migrate:staging`
- [auth 模块] 规则：不允许跳过测试 → 下次执行：auth 相关改动后必须跑 `npm test -- auth`
```

### decisions.log.md 格式

每条不超过 5 行，格式：

```text
- [决策类型] 决策内容：xxx
  影响：xxx
  依据：xxx（用户确认/踩坑验证/新约定）
  日期：yyyy-mm-dd
```

示例：

```text
- [架构决策] 认证状态存储改为 httpOnly cookie
  影响：所有 auth 相关任务，不再使用 localStorage 存登录状态
  依据：用户确认（档位 C 澄清，安全风险）
  日期：2026-07-05
```

## 膨胀控制与归档

### lessons.md 归档规则

1. `lessons.md` 超 50 条时，触发归档。
2. 归档操作：将最旧的 20 条移到 `.align/archive/lessons-{yyyy-mm-dd}.md`。
3. 归档文件不再被 agent 自动读取（只有 `lessons.md` 在读取顺序中）。
4. 归档后 `lessons.md` 保留最新 30 条。
5. 归档操作在门 5 沉淀门执行时自动完成，不需要用户干预。

### spec.md 膨胀控制

1. `spec.md` 超 150 行时，精简或拆分。
2. 拆分策略：将背景性信息移到 `context.md`，将旧决策移到 `decisions.log.md`。
3. 精简策略：合并重复规则，删除已过时条款。

### context.md 膨胀控制

1. `context.md` 超 100 行时，精简或归档。
2. 归档策略：将旧架构决策移到 `decisions.log.md`。
3. 精简策略：合并重复上下文，删除已过时的项目阶段描述。

### decisions.log.md 膨胀控制

1. `decisions.log.md` 超 100 条时，将最旧 40 条归档到 `.align/archive/decisions-{yyyy-mm-dd}.md`。
2. 归档文件不再被 agent 自动读取。

## 正例：什么该沉淀

### 正例 1：踩坑 → 规则

```text
- [API 修改] 规则：修改 public API 必须同步更新 TypeScript 类型定义 → 下次执行：改 API 后跑 `tsc --noEmit` 验证类型
```

**为什么该沉淀**：下次任何 agent 修改 API 时，读到这条规则会自动跑类型检查，避免重复踩坑。

### 正例 2：用户纠正 → 边界

```text
- [数据库操作] 规则：禁止直接执行 DROP 语句 → 下次执行：涉及删除表/列时先确认备份
```

**为什么该沉淀**：用户纠正了危险操作，下次 agent 读到这条规则会先确认备份，避免数据丢失。

### 正例 3：推翻假设 → 验证

```text
- [部署环境] 规则：生产环境使用 MySQL 非 PostgreSQL → 下次执行：涉及 SQL 语法时按 MySQL 兼容写
```

**为什么该沉淀**：假设被证伪，下次 agent 读到这条规则会用正确的 SQL 方言，避免语法错误。

---

## 反例：什么不该沉淀

### 反例 1：空洞总结

```text
- 本次重构任务顺利完成，代码质量良好。
```

**为什么不该沉淀**：删掉这条，下次 agent 的行为不会改变。没有场景、没有规则、没有执行动作。

### 反例 2：一次性信息

```text
- 修改了 src/config/parseConfig.js 第 42 行，把 if-else 改成 switch。
```

**为什么不该沉淀**：具体行号和改法只对本次任务有效，下次任何任务都不会复用这个信息。

### 反例 3：不能改变行为的记录

```text
- 用户觉得这个功能应该放在 v2 版本做。
```

**为什么不该沉淀**：这是产品优先级决策，不是 agent 执行规则。无法写成"遇到 X 场景 → 执行 Y"。如果确实是重要决策，应写入 `decisions.log.md` 而非 `lessons.md`。

## 与 R10 沉淀门的关系

`04-transform-rules.md` 中 R10 沉淀门仍是转换规则的一部分。本文件定义 v3 项目运行时中的自动沉淀目标、格式和膨胀控制。

| 属性 | R10 沉淀门（04-transform-rules.md） | 本文件（07-precipitation.md） |
| --- | --- | --- |
| 触发条件 | 长期项目、代码库任务、团队规则、重复性任务 | 踩坑/用户纠正/新约定/推翻假设/高风险决策确认 |
| 输出 | 要求 agent 输出"应沉淀的项目规则/术语/风险/模板" | 定义写入 `.align/lessons.md` 和 `.align/decisions.log.md` 的精确格式 |
| 禁止行为 | 不得输出"本次任务很顺利"式空洞沉淀 | 不得写入空洞总结、一次性信息、不能改变行为的记录 |
| 关系 | R10 是转换规则中的沉淀触发器 | 本文件是 R10 在 v3 项目运行时中的落地实现 |

两者不冲突：R10 触发沉淀，本文件定义沉淀的格式、目标和膨胀控制。

## 与门 5 沉淀门的关系

`06-lifecycle-gates.md` 中门 5 沉淀门是生命周期五门的最后一门。本文件定义门 5 的详细执行协议。

| 属性 | 门 5 沉淀门（06-lifecycle-gates.md） | 本文件（07-precipitation.md） |
| --- | --- | --- |
| 触发时机 | 任务结束后 | 同 |
| 执行动作 | 把经验写入 `.align/lessons.md` 或 `.align/decisions.log.md` | 本文件定义写入格式和判定标准 |
| 静默度 | 无可沉淀内容则完全静默；有沉淀时回复末尾用一行告知 | 同 |
| 跳过条件 | 无触发条件 / `.align/` 不存在 / `[直出]` | 本文件补充：无触发条件 = 无踩坑/纠正/新约定/推翻假设/高风险决策 |

## 与 `.align/` 四件套模板的引用关系

`.align/` 四件套模板已在 `core/templates/` 定义：

- `core/templates/ALIGN-SPEC.md` → 生成 `.align/spec.md`
- `core/templates/ALIGN-CONTEXT.md` → 生成 `.align/context.md`
- `core/templates/ALIGN-LESSONS.md` → 生成 `.align/lessons.md`
- `core/templates/ALIGN-DECISIONS.md` → 生成 `.align/decisions.log.md`

本文件定义 `lessons.md` 和 `decisions.log.md` 的写入格式和膨胀控制，模板定义初始结构和填写标准。两者配合：模板管"长什么样"，本文件管"怎么往里加"。
