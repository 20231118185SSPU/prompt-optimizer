# 转换规则 R1-R10

> 协议内核 SSOT 文件。内容由 `build/` 脚本生成到 `dist/`，禁止手改 `dist/`。

## 6. 转换规则



核心原则：信息不足时，宁可停下来澄清，也不要猜测和妥协。转换规则不是润色技巧，而是把原始请求补成可执行契约的强制执行清单。

本节即零妥协转换（Zero-Compromise Transformation）：所有强制规则、禁止行为和验证门都不得弱化为建议。



### 转换规则执行清单（Zero-Compromise Transformation）



执行前必须自检：



- [ ] 意图探查已完成（如触发）。

- [ ] 五维诊断所有维度≥1分。

- [ ] D5验证性≥1分（强制）。

- [ ] 高风险信号已处理（如有）。

- [ ] 用户已确认真实意图（如有歧义）。



未满足以上任一项时，不得输出最终优化结果；必须先澄清、探查或补全到可执行状态。



### R1 精确性注入（强制执行）



触发：检测到模糊动词（优化、改进、处理、看看、弄一下、帮我、做个、加个、写个、搞个、弄个、实现个、建个等创建类动词）。



禁止行为：



- 不得用"可能""也许""建议"等词替代明确目标。

- 不得自行猜测用户意图。

- 不得用"根据常规做法"替代用户确认或可验证上下文。



操作：替换为"精确动词 + 具体对象 + 目标结果"。如果对象或目标缺失且会改变结果，先问一个问题；如果可从文件、文档、日志、URL获得，写入 agent 的感知步骤。



示例：



- 原始：帮我优化这个项目

- 优化：审计当前项目的文档结构和 prompt 转换流程，找出导致 agent 误解用户意图的缺口，并重构核心方法论和模板。



### R2 边界锁定（强制执行）



触发：缺少范围、权限、数量、排除项。



禁止行为：



- 不得用"尽量不改"替代"禁止修改"。

- 不得用"一般来说"替代明确权限。

- 不得假设用户"应该允许"某些修改。



操作：补充"只做什么 / 不做什么 / 需要确认后才能做什么"。涉及文件修改、联网、删除、提交、部署、外部系统写入时，必须写清权限边界。



### R3 交付物锁定（强制执行）



触发：未指定输出格式。



禁止行为：



- 不得用"灵活输出"替代明确格式。

- 不得让 AI 自行决定详细程度。



操作：指定最终交付形态、结构和篇幅，如 Agent Brief、PRD、代码补丁、审查报告、决策矩阵、JSON。详细程度必须以"简要/完整/逐项/限定字数"等可执行方式表达。



### R4 上下文锚定（强制执行）



触发：缺少背景、受众、用途、项目状态。



禁止行为：



- 不得假设"用户肯定知道自己的技术栈"。

- 不得省略"显而易见"的背景。



操作：补充最小必要背景，包括用户场景、当前状态、受众、用途。可由 agent 自行读取的上下文必须写成"先读取 X"，不可转嫁给用户。



### R5 反面约束（强制执行）



触发：无禁止项说明，或 agent 自由度过大。



禁止行为：



- 不得用"注意"替代"禁止"。

- 不得用"建议"替代"必须"。



操作：添加 1-3 条最关键的"不要做 / 禁止 / 必须保持"。优先限制最容易造成损失的行为，并从 `core/templates/ANTI-PATTERNS-REFERENCE.md` 选择任务类型匹配的反面约束。



### R6 范围裁剪（强制执行）



触发：一个指令包含3个以上独立任务，或任务之间目标、交付物、验收方式明显不同。



禁止行为：



- 不得试图"一次性完成所有任务"。

- 不得擅自决定哪些任务可以合并。



操作：拆成主任务、次任务、后续任务。当前只让 agent 处理一个主目标；合并或改优先级会改变结果时，先问一个问题并给推荐答案。



### R7 示例嵌入（条件执行）



触发：主观任务、风格任务、复杂格式任务没有例子。



禁止行为：



- 不得用"参考类似风格"替代具体示例。



操作：给 1 个正例，必要时给 1 个反例。示例必须服务验收和边界，不得引入新的目标或替用户决定方向。



### R8 验证门（强制执行，最高优先级）



触发：D5=0 或无明确验收标准。



禁止行为：



- 不得使用抽象验收标准（"要好用""要稳定"）。

- 不得省略验收标准（即使用户未提及）。

- 不得使用不可执行的标准（"符合最佳实践"）。



操作：按任务类型从 `references/acceptance-checklist.md` 选择可执行验收项，添加可检查的验收项、自检项、测试命令、量化指标或人工检查清单。



最小清单：



- 代码类：测试命令、类型检查、代码质量、新依赖说明、public API 变更标记。

- 调研类：覆盖维度、来源依据、明确推荐、场景匹配。

- 写作类：篇幅、语气、结构、术语准确性。

- 重构类：现有测试、兼容性、复杂度或重复改善、修改范围和风险说明。

- 部署类：部署前检查、环境配置、回滚条件、监控日志、版本和影响范围。



### R9 意图路由（强制执行）



触发：所有任务。



必须写入每个 Agent Brief：



```markdown

## 执行策略



- 信息足够且低风险 → 直接执行。

- 关键信息缺失且会改变目标 → 停下来，只问一个问题，附推荐答案。

- 信息可从文件/文档/日志/URL获得 → 自己读取，不问用户。

- 高风险操作（删除/生产/不可逆）→ 先输出方案和影响面，等待确认。

- 只读分析任务 → 禁止修改任何文件。

- 代码修改任务 → 先读上下文，再最小变更。

```



禁止行为：



- 不得省略"信息不足时怎么办"的指引。

- 不得让 agent 自行决定是否需要确认高风险操作。



### R10 沉淀门（条件执行）



触发：长期项目、代码库任务、团队规则、重复性任务。



禁止行为：



- 不得输出"本次任务很顺利"式的空洞沉淀。

- 每条沉淀必须能改变下次 agent 的行为，否则删掉。



操作：要求 agent 在完成后输出"应沉淀的项目规则 / 术语 / 风险 / 模板"。沉淀项必须包含适用场景、触发条件和下次执行规则。
