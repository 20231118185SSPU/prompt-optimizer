# 从零项目访谈协议

> 本文件是 `align-init --new` 对从零项目进行规范访谈的决策树。
> 目标：4 个问题完成接入，每问附推荐答案，用户可以连按四次"就按推荐的"。

## 访谈流程

```text
/align-init --new
→ 走 spec-kit/interview.md 决策树：
   Q1 项目一句话目标 → Q2 技术栈选型（给推荐+理由）→
   Q3 质量门槛（测试策略/验证命令）→ Q4 高风险边界
→ 每问附推荐答案，用户可以连按四次"就按推荐的"
→ 生成 .align/ 四件套 + 项目骨架建议 + 注入挂载区
```

---

## Q1：项目一句话目标

### 问题

```text
你的项目要解决什么问题？用一句话描述。

推荐答案：为 [目标用户] 提供 [核心功能]，解决 [具体痛点]。
```

### 推荐答案示例

```text
为前端开发者提供 prompt 优化工具，解决 AI agent 误解用户意图的问题。
```

### 判定标准

- 用户回答必须包含：目标用户 + 核心功能 + 具体痛点。
- 如果用户回答过于模糊（如"做一个好用的工具"），追问"这个工具给谁用？解决什么痛点？"
- 回答写入 `.align/context.md` 的"项目目标与当前阶段"。

### 跳过条件

无。这是必答问题，是后续所有规范的出发点。

---

## Q2：技术栈选型

### 问题

```text
你打算用什么技术栈？

推荐答案：根据项目类型推荐：
- Web 应用：React + TypeScript + Vite
- API 服务：Node.js + TypeScript + Fastify
- CLI 工具：Node.js + TypeScript + Commander
- 数据分析：Python + Jupyter + pandas
- 通用：TypeScript（语言）+ pnpm（包管理）+ Vitest（测试）+ ESLint（lint）
```

### 推荐答案理由

- **TypeScript**：类型安全，IDE 支持好，生态成熟。
- **pnpm**：速度快，磁盘高效，monorepo 友好。
- **Vitest**：与 Vite 生态兼容，API 类似 Jest 但更快。
- **ESLint**：可配置性强，插件生态丰富。

### 判定标准

- 用户回答必须包含：语言 + 框架 + 包管理器。
- 如果用户说"用最新版的"，追问具体框架名和版本号。
- 回答写入 `.align/spec.md` 的"技术栈与版本"。

### 跳过条件

无。技术栈决定了后续所有工具链配置。

---

## Q3：质量门槛

### 问题

```text
你对代码质量有什么要求？

推荐答案：
- 测试策略：关键逻辑必须有单元测试，覆盖率目标 ≥70%
- 验证命令：`pnpm test`（单元测试）+ `tsc --noEmit`（类型检查）+ `pnpm run lint`（lint）
- 提交前必须跑：`pnpm test && tsc --noEmit && pnpm run lint`
- CI 流程：push 时自动跑 lint + test + build
```

### 推荐答案理由

- **覆盖率 70%**：够用但不浪费精力在覆盖率数字上。
- **三个命令**：测试、类型、lint 是最低必要门槛。
- **CI 自动化**：减少人工遗忘风险。

### 判定标准

- 用户回答必须包含：测试策略 + 至少一个验证命令。
- 如果用户说"先不写测试"，记录为高风险项（无测试 = 验收靠人工）。
- 回答写入 `.align/spec.md` 的"测试与验证命令"。

### 跳过条件

无。质量门槛是门 4 验证门的执行依据。

---

## Q4：高风险边界

### 问题

```text
这个项目里有哪些操作是你认为高风险的？

推荐答案：
- 数据库 schema 变更（必须先在 staging 验证）
- 生产环境部署（必须先确认版本号和变更清单）
- 删除文件/目录（必须先确认引用）
- 引入新依赖（必须确认包体积和许可证）
- 修改 public API（必须同步更新调用方）
```

### 推荐答案理由

以上五项是通用高风险操作，适用于大多数软件项目。项目类型不同可能需要补充：

- 有认证的项目：修改认证/权限逻辑
- 有支付的项目：修改支付流程
- 有数据迁移的项目：执行数据迁移脚本

### 判定标准

- 用户回答必须包含至少 3 项高风险操作。
- 如果用户说"没有高风险操作"，推荐通用五项并问是否采用。
- 回答写入 `.align/spec.md` 的"高风险操作清单"。

### 跳过条件

无。高风险清单是三档路由安全阀的判断依据。

## 访谈完成后的输出

访谈完成后，生成以下文件：

1. **`.align/spec.md`**：从 `spec-sections/` 选择匹配的预设，填入用户回答。每条标注 `[原文]`（用户明确说的）。
2. **`.align/context.md`**：填入 Q1 的项目目标和当前阶段（"v0.1 从零开发中"）。
3. **`.align/lessons.md`**：初始为空。
4. **`.align/decisions.log.md`**：记录访谈中的关键决策（如技术栈选型理由）。

同时输出项目骨架建议：

```text
建议项目结构：
├── src/
│   ├── features/        # 功能模块
│   ├── shared/          # 共享工具
│   └── config/          # 配置
├── tests/               # 集成测试
├── .align/              # 对齐运行时
├── CLAUDE.md            # 挂载区
└── package.json
```

---

## 与 spec-sections/ 的关系

访谈回答与 `spec-sections/` 章节库配合使用：

- `spec-sections/` 提供 2-3 个可选预设（如技术栈有"React+TS"、"Vue+TS"、"Node+TS"三个预设）。
- 用户回答 Q2 后，选择最接近的预设，填入用户的具体版本号。
- 如果用户回答不匹配任何预设，使用通用预设并标注 `[原文]`。

`spec-sections/` 保证生成的规范结构一致、可判定（每条规范必须写成"可验证的行为"）。
