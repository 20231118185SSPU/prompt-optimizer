# v3.2.0 稳定版改进与优化执行方案

> 状态：执行中；S0-S2 已完成，等待进入 S3。
> 日期：2026-07-12。
> 基线版本：`v3.2.0-rc.1`。
> 目标版本：`v3.2.0`。
> 核心原则：先统一决策语义和评测可靠性，再补发布证据；不以本地双 provider 凭据作为稳定版前提。

## 1. Agent Brief

### 1.1 目标

将当前候选版收敛为可维护、可验证、可诚实发布的稳定版，重点解决以下问题：

1. TypeScript runtime、legacy router、shell hook 和协议文案可能对同一请求给出不同路由。
2. 模型返回空输出时，单条记录虽然失败，但评测批次仍可能以退出码 0 结束，导致上层误判和重复执行。
3. 同步 CLI 调用期间缺少进度与恢复机制，容易表现为长时间空白或重复轮询。
4. Claude/Codex 完整三臂测试依赖本地凭据、CLI 状态和付费调用，不适合作为开发者日常门槛。
5. `.align/debt.md` 中仍有稳定版前证据债务和 Windows 清理债务。

### 1.2 信息来源与置信度

- `[原文]`：用户要求解决空回复后反复执行的问题。
- `[原文]`：用户认为在本地完成 Claude/Codex 三臂实际测试存在困难。
- `[原文]`：项目当前处于 `v3.2.0-rc.1` 候选阶段。
- `[原文]`：公共机器 route 为 `pass / enrich / clarify / block`。
- `[推断]`：Alignment Decision 应成为所有宿主共享的唯一决策 Interface。
- `[推断]`：真实模型证据与产品正确性 Gate 应分级，不能因缺少某个 provider 凭据阻塞全部稳定版工作。
- `[假设]`：本轮不新增生态 handoff、产品功能或新的 runtime 依赖。
- `[假设]`：真实模型调用只有在用户单独批准 provider、模型、样本和预算后才执行。

### 1.3 范围

包含：

- 统一路由 SSOT 和宿主 Adapter 投影。
- 加固真实模型评测 runner。
- 增加远程、手动、受预算约束的评测入口。
- 冻结稳定版证据政策并重分类债务。
- 完成 fresh blind evidence、发布文档、版本同步和最终审核。

不包含：

- 新增 OpenSpec、Superpowers、ECC 等 handoff。
- 自动运行付费模型。
- 自动 push、tag、GitHub Release 或 npm publish。
- 删除 legacy `.align/context.md`。
- 在没有兼容性必要时修改 Alignment Decision v1 schema。
- 手工修改 `dist/`。

## 2. 冻结决策

### 2.1 唯一决策 Interface

稳定版目标 Interface：

```typescript
align(request, projectContext, hostCapabilities): AlignmentDecision
```

规则：

- `classifier` 只产生内部信号，不再拥有独立的宿主路由语义。
- `HIGH / VAGUE / GRAY / CLEAR` 只能作为兼容展示标签，必须由 Alignment Decision 投影得到。
- 所有执行、澄清、等待确认和停止行为只读取 `route`、`reasons` 与 `next.action`。
- Claude Code、Codex、Cursor 和 shell fallback 是 Adapter，不得各自重新决定 route。
- `risk.*` 表示必须经过安全路由，不等于永久阻断。
- 信息不足时 `clarify`；授权、政策或 baseline 阻断时 `block`；授权和恢复条件完整时允许 `enrich`。

### 2.2 稳定版证据政策

| 证据 | 稳定版要求 | 说明 |
| --- | --- | --- |
| E2 确定性 corpus/schema/golden | 必须 | 每次 PR 和发布前运行 |
| E3 安装器/doctor/Adapter 沙箱 | 必须 | 覆盖 Bash、PowerShell、Claude Code、Codex |
| Fresh post-fix blind review | 必须 | 新语料先冻结，运行后立即标记 consumed |
| 单一固定模型三臂 | 推荐 | 有远程凭据和预算时运行，不要求开发者本地完成 |
| 第二 provider 完整三臂 | 非阻塞 | 未完成时禁止声明对应 E5 能力 |
| 真实首轮成功率/返工轮数 | 发布后持续收集 | 不集中消耗额外模型预算 |

稳定版允许在没有 Claude/Codex 双 provider E5 的情况下发布，但支持矩阵必须明确标注证据等级。缺少 E5 只能限制宣传口径，不能伪装为已经完成。

### 2.3 付费与重试政策

- 默认不重试模型调用。
- 空输出、非法 JSON、schema 错误和凭据错误均计为真实失败，不得静默重跑。
- 只有显式 `--retry-failures 1` 才允许补跑一次。
- 每个远程任务必须声明 provider、实际模型、case 数、repeat 和预算上限。
- 达到预算或调用数上限时立即停止，并保留已完成 evidence。
- 不得因单条失败清空整个输出文件或重新执行已完成 case。

## 6. S2：评测 Runner 可靠性

### 主要文件

- `tests/eval/run-eval.js`
- `tests/eval/score-eval.js`
- 新增 runner 单元/集成测试和 fake CLI fixtures

### 失败分类

```text
empty_output
missing_output_file
invalid_json
schema_mismatch
cli_exit_nonzero
timeout
credential_error
budget_exceeded
cleanup_error
```

### 任务

- `S2-01`：退出码 0 但空输出必须记录 `empty_output`。
- `S2-02`：任意 case 失败时批次默认返回非零退出码；提供显式 `--allow-failures` 供证据采集使用。
- `S2-03`：增加 `--resume`，以 provider/model/arm/case/iteration 为稳定键跳过已完成记录。
- `S2-04`：增加 `--retry-failures 0|1`，默认 0，禁止无限重试。
- `S2-05`：增加 `--max-cases`、`--max-cost-usd` 和 `--timeout-ms`。
- `S2-06`：每次调用前输出 START，结束后输出状态、耗时、用量和失败分类。
- `S2-07`：将模型进程改为可观察的异步执行，长调用定期输出 heartbeat；不得启动第二个相同调用。
- `S2-08`：不再启动时清空已有 evidence；新运行写入新 runId 或显式覆盖文件。
- `S2-09`：Windows cleanup 使用独立重试和最终路径记录，不得改变模型调用状态。
- `S2-10`：score 报告区分 completed、model failure、runner failure 和 cleanup warning。

### Gate

- fake CLI 退出 0 且无输出时，case 为 `failed/empty_output`，批次退出非零。
- fake CLI 写入非法 JSON 时，失败类型为 `invalid_json`。
- `--resume` 不重复调用已经完成的 case。
- cleanup failure 不触发模型重试。
- 达到预算后不再启动新调用，已有 JSONL 保持可解析。
- 测试全程不调用真实模型。

### 执行记录（2026-07-13）

| 项目 | 结果 |
| --- | --- |
| 失败关闭 | 覆盖 `empty_output`、`missing_output_file`、`invalid_json`、`schema_mismatch`、`cli_exit_nonzero`、`timeout`、`credential_error`、`budget_exceeded` 和 cleanup warning；case 失败时批次默认非零，`--allow-failures` 可显式放行 |
| 恢复与重试 | `--resume` 按 provider/model/arm/case/iteration 稳定键跳过已完成记录；`--retry-failures` 只允许 `0` 或 `1`，默认不重试 |
| 预算与可观察性 | `--max-cases`、`--max-cost-usd`、`--timeout-ms` 生效；有限成本预算必须同时声明保守的 `--max-cost-per-call-usd`，runner 在启动调用前预留该上界，缺少上界时失败关闭；每次调用输出 START、RESULT、latency、usage、failure，长调用输出 heartbeat；同一稳定键保持单一 active call |
| Evidence | 默认追加并保留已有 JSONL；只有 `--overwrite` 显式清空；预算停止时保留已完成记录并追加可解析的 `budget_exceeded` 记录 |
| Cleanup | 临时目录独立重试三次；最终失败只写 `cleanupWarning`、错误和脱敏路径，不改变模型 status，也不触发模型重试 |
| Scorer | 报告分别统计 `completed`、`modelFailures`、`runnerFailures` 和 `cleanupWarnings`，质量指标只使用 completed 记录 |
| 验证 | fake CLI 集成矩阵、scorer 分类、证据隐私、既有 held-out/remediation 与 56 条冻结行为语料均完成本地验证；未调用真实模型 |

S2 Gate 结论：通过。旧 held-out raw Gate 的历史失败仍按原 evidence 保留，不作为本次 runner 回归失败，也未被重标为 fresh evidence。

## 7. S3：远程手动三臂评测

### 目标

把真实模型调用从开发者本地必做项改为可审计、可预算、手动触发的远程任务。

### 任务

- `S3-01`：新增手动 workflow，输入 provider、model、arm、corpus、ids、repeat 和预算。
- `S3-02`：workflow 启动前验证 secret 是否存在；缺失时安全失败，不调用模型。
- `S3-03`：设置 concurrency，禁止同一 provider/model/corpus 重复并发。
- `S3-04`：设置任务超时和调用上限。
- `S3-05`：evidence 脱敏后只上传 artifact，不自动 commit。
- `S3-06`：记录请求模型和 provider 返回的实际模型；两者不一致时标记 `model_identity_mismatch`。
- `S3-07`：CLI 宿主接线测试与模型行为测试分离，避免把代理模型结果误称为 Claude/Codex 模型证据。
- `S3-08`：提供只运行 1 个 case、1 个 arm 的 smoke 模式。

### Gate

- workflow 只能手动触发。
- 日志和 artifact 不包含 API key、token、session id、本地用户路径或原始敏感上下文。
- 无凭据、超预算或模型身份不明时，结果不得标为有效 E5。
- workflow 不 push、不 tag、不发布。

## 8. S4：Fresh 稳定版证据

### 任务

- `S4-01`：创建全新 blind corpus，必须在首次 runtime 执行前冻结 hash 和 manifest。
- `S4-02`：覆盖 pass、context enrich、低分 clarify、高风险缺信息、授权 block、完整授权 enrich、只读反例和 XY Problem。
- `S4-03`：运行确定性 runtime，生成只含请求、fixture 和 proposed behavior 的 blind input。
- `S4-04`：由未查看 oracle/实现的独立 reviewer 按固定 rubric 评审。
- `S4-05`：评审完成后立即把 manifest 标记为 consumed。
- `S4-06`：如需修复，只能把该 corpus 作为 regression，禁止重新标记 held-out。
- `S4-07`：有远程预算时，对一个固定真实模型运行三臂；无预算时保留为未完成 E5，不伪造 Gate。

### 稳定版硬 Gate

- 高风险漏放率：`0%`。
- 不必要澄清率：`<=10%`。
- route 与独立评审意见一致，或每个有意偏离都有书面解释。
- clarify 恰好一个最高价值问题并附推荐答案。
- pass/enrich 的验收可执行或可明确人工判定。
- 不替用户决定产品方向、架构优先级或不可逆操作。
- frozen、blind、review、summary、runtime bundle hash 和隐私检查证据齐全。

## 13. 风险与回滚

| 风险 | 预防 | 回滚条件 |
| --- | --- | --- |
| 路由统一造成旧宿主行为变化 | 保留兼容投影和 golden conformance | 任一既有安全 case 漏放立即回滚该波次 |
| Runner 改造损坏历史 evidence | 新旧格式 reader 回归、禁止覆盖旧文件 | 旧 evidence 无法验证时停止 S3 |
| 远程调用失控 | 手动触发、预算、并发锁、零默认重试 | 超预算或重复调用立即禁用 workflow |
| 模型身份不一致 | 同时记录 requested/resolved model | 无法确认模型时降级为无效 E5 |
| 文档与 runtime 再次漂移 | route 文案由 decision/policy 投影或机械验证 | 发现冲突时不得发布 |
| 稳定版范围膨胀 | 冻结非目标和独立提交 | 新生态/新依赖移到后续版本 |
