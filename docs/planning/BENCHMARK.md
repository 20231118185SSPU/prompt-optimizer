# Prompt Optimizer 回测基准

> P3 验证与打磨：用 10 个真实指令跑完整流程，记录规则推演结果。本文不调用外部模型，只用当前 `METHODOLOGY.md`、`TRANSFORM.md` 和 `SKILL.md` 的规则做预期判定。

## 基准目标

验证升级后的 Agent 意图对齐器是否同时满足两类要求：

1. 严格性：高风险、目标不清、验收缺失、意图偏差类请求必须先对齐，不得直接输出最终优化结果。
2. 可用性：简单、低风险、信息完整或 `[直出]` 请求不应被过度澄清。

## P3 验证与打磨

原路线图基准：

10. 用 10 个真实指令跑完整流程，记录进 `BENCHMARK.md`。
11. 检查 skill 在 Claude Code 中的实际触发和输出质量。
12. 根据实测结果调整规则的严格度，防止过度澄清骚扰用户。

本会话只完成第 10 项：建立 benchmark 并按当前规则推演结果。第 11、12 项保留给后续会话。

## Case 01：高抽象动词

**原始指令**

```text
帮我优化一下这个项目。
```

**预期路由**

Clarify First。

**触发规则**

- Step 0 意图探查触发：高抽象动词“优化”。
- D1=0、D2=0、D3=0、D4=0、D5=0，总分<6。
- D5=0，必须补全验收标准。

**预期输出类型**

当前理解 + 最大不确定点 + 一个问题 + 推荐答案 + 共识快照。

**是否应澄清**

是。目标、范围和验收全部缺失，直接优化会替用户决定方向。

**验收点**

- 只问一个问题：这次优化最终要解决什么问题。
- 推荐答案应优先选择“降低 agent 误解用户意图”这类与项目定位一致的目标。
- 不输出最终 Agent Brief。

## Case 02：模糊目标

**原始指令**

```text
让这个接口更稳定。
```

**预期路由**

Intent Probing。

**触发规则**

- Step 0 意图探查触发：模糊目标“更稳定”。
- D1=0，目标不可量化。
- D5=0，缺少可执行验收。

**预期输出类型**

意图探查问题或 Clarify First 输出。

**是否应澄清**

是。稳定可能指错误率、超时、重试、并发、数据一致性或降级策略，猜错会改变方案。

**验收点**

- 必须问“最终要解决什么稳定性问题”或等价问题。
- 推荐答案应指向可量化指标，例如“降低 500 错误率或超时率”。
- 不直接补成某个技术方案。

## Case 03：缺少 WHY

**原始指令**

```text
把这个函数拆成三个小函数。
```

**预期路由**

Intent Probing。

**触发规则**

- Step 0 意图探查触发：只说做什么，不说为什么。
- 意图-表达偏差风险：用户指定了结构动作，但未说明原始问题。
- D5=0，缺少验收标准。

**预期输出类型**

一个意图确认问题 + 推荐答案 + 共识快照。

**是否应澄清**

是。拆分函数可能是为了降低复杂度、提高测试性、满足 review 或教学解释，不同 WHY 会改变验收。

**验收点**

- 只能问一个最高价值问题：“这次拆分最终要解决什么问题？”
- 推荐答案可以是“降低圈复杂度并保持行为不变”。
- 不直接规定拆成哪三个函数。

## Case 04：隐含假设

**原始指令**

```text
帮我把登录状态存到 localStorage，这样刷新页面就不会掉登录了。
```

**预期路由**

Intent Probing。

**触发规则**

- Step 0 意图探查触发：用户预设解决方案 localStorage，但未说明认证机制和安全约束。
- 可能涉及安全、权限和数据持久化。
- R2 边界锁定、R5 反面约束、R8 验证门。

**预期输出类型**

方案确认式澄清。

**是否应澄清**

是。认证状态存储位置会影响安全模型；猜错成本高于一次澄清成本。

**验收点**

- 不能直接接受“localStorage”作为已确认最佳方案。
- 应询问是否必须使用 localStorage，还是目标只是刷新后保持登录。
- 推荐答案应偏向先确认现有认证方案，再选择存储方式。

## Case 05：XY Problem

**原始指令**

```text
帮我用正则解析这个 HTML 提取商品价格。
```

**预期路由**

Clarify First。

**触发规则**

- 意图-表达偏差：解决方案伪装成需求（XY Problem）。
- 用户要求正则，但真实目标可能是稳定提取价格。
- D5=0，缺少成功标准。

**预期输出类型**

**验收点**

- 必须包含：`我可以按你说的做【X】。不过如果你的最终目的是【推测的Y】，【Z方案】通常更合适。请确认：A. 就按X做 B. 改用Z做`
- 必须把真实目标标为推测或待确认。
- 不直接拒绝用户方案。

## Case 06：症状伪装成原因

**原始指令**

```text
这个页面加载慢，帮我加一个 loading 动画。
```

**预期路由**

Agent Brief。

**触发规则**

- 意图-表达偏差：症状伪装成原因。
- 表面动作是加 loading，真实问题可能是接口慢、渲染慢或资源阻塞。
- R9 执行策略要求先诊断根因再决定修复层次。

**预期输出类型**

Agent Brief，背景中记录症状和可能根因，工作方式要求先诊断。

**是否应澄清**

否。可以在不问用户的情况下要求未来 agent 先读取性能数据、网络面板或代码路径；但如果要只做 loading 而不诊断，应让 agent 执行时确认。

**验收点**

- 背景同时写明“页面加载慢”和“加 loading 只是可能方案”。
- 工作方式必须包含“先诊断根因再决定修复层次”。
- 验收标准不能只有“有 loading”，还应包含加载状态、错误状态和性能线索记录。

## Case 09：`[直出]` 任务


Direct Output。

## 待调整项记录

当前仅按规则推演，未调用外部模型，因此不直接修改核心规则。后续会话实测时重点观察：

1. Case 01 和 Case 02 是否因为“高抽象动词”过度频繁进入澄清；如果用户同时提供了目标、范围和验收，应按跳过条件放行。
2. Case 06 是否能稳定区分“可以先诊断”的场景和“必须问用户是否只做 loading”的场景。
3. Case 09 是否在 `[直出]` 下仍保留验收门，但不输出外壳诊断。
4. Case 10 是否会因为“重构”被误判为高风险并强制意图探查；信息完整时应跳过探查。

## 本轮结论

10 个 benchmark case 已覆盖 P3-1 要求的 10 类场景。当前规则的关键风险不是严格性不足，而是简单任务和信息完整任务可能被“高抽象动词”或“重构”关键词误触发澄清；后续应通过实际 skill 触发测试确认是否需要微调。

## P3-3 最终一致性验收报告

### 严格度微调结论

已根据上述待调整项做最小规则调参：严格性总原则明确为 **澄清的成本 < 猜错的返工成本时才澄清**。`重构`、`提升性能`、高抽象动词不再被解释为无条件打断用户的理由；只有缺少目标、范围、约束或验收，且猜错会改变执行结果时，才进入意图探查或澄清访谈。

### P0/P1/P2/P3 覆盖情况

- P0 零容忍诊断和验证门：已覆盖 D1-D5 严格评分、D5=0 强制补全、四条硬性重做门槛和验收清单库引用。
- P1 意图探查和偏差检测：已覆盖意图探查协议、跳过条件、XY Problem、症状伪装成原因、局部视角遮蔽全局目标。
- P2 共识与回验：已覆盖共识快照、契约回验四问、置信度 `[原文]` / `[推断]` / `[假设]` 标注，以及 `[假设]≤2` 允许先输出但必须验证。
- P3 验证与打磨：已用 10 个 benchmark case 覆盖严格性和可用性，并针对过度澄清风险完成规则微调。

### 过度澄清风险判断

当前仍存在低到中等风险：模型可能看到“优化”“重构”等词后过早进入澄清。已通过三条规则对冲：

1. 信息完整、低风险、目标单一或 `[直出]` 时跳过意图探查。
2. 简单任务走基础优化，不套重型 Agent Brief。
3. `[假设]` 不超过 2 条时允许先输出并标注待确认，不强制中断。

### 后续建议观察的 case

- Case 01/02：用户用了高抽象动词，但同时给出目标、范围和验收时，是否能放行。
- Case 06：症状伪装成原因时，是否能先要求 agent 诊断，而不是立刻追问。
- Case 09：`[直出]` 是否稳定只输出正文，同时保留内部验收门。
- Case 10：信息完整的 `重构` 是否走 Basic Optimize 或 Agent Brief，而不是强制意图探查。

### 最终总结

这次升级的本质：从"把话说清楚的工具"升级为"确认你真正想要什么的协议"。

三个不可妥协的支柱：

1. **先探意图，再谈优化** — 用户的第一句话是线索，不是需求。
2. **补结构不补方向** — 缺的格式可以补，缺的决策必须问。
3. **无验收，不交付** — 每份契约必须自带"怎么算做对了"。
