# Agent 意图对齐方法论

> 目标：让 AI agent 精准理解用户真正想要什么，并用可验证、可迭代、可沉淀的方式执行。

---

## 1. 核心判断

Prompt 优化的重点不是“写得更漂亮”，而是降低三类偏差：

1. **意图偏差**：用户想要 A，agent 理解成 B。
2. **范围偏差**：用户只想改一处，agent 扩大到整套方案。
3. **验收偏差**：agent 以为完成了，用户发现不符合实际目标。

因此，一个高质量 prompt 应该像一份轻量任务契约，而不是一句请求。

严格性有代价：**澄清的成本 < 猜错的返工成本时才澄清**。如果任务信息完整、低风险、目标单一，或用户明确使用 `[直出]`，不得为了形式完整而打断用户；应走基础优化或直出路径，并把必要的验收和假设验证写入任务契约。

---

## 阶段0：意图探查（Intent Probing）

**问题**：用户表达的需求往往是表层症状，不是根本目标。

**案例**：

- 用户说："帮我优化这个函数的性能"
- 真实意图可能是：
  - A. 这个函数在生产环境卡顿，需要降低延迟
  - B. 代码审查要求圈复杂度<10，需要重构
  - C. 想学习性能优化技巧，需要教学式解释
  - D. 看到网上说某种写法更快，想验证

**解决方案**：在五维诊断之前增加**意图探查层**。意图探查不是默认阻塞所有请求；只有表面需求和真实意图可能不一致时才触发。

### 意图探查协议（Intent Probing Protocol）

#### 触发条件

当用户指令包含以下信号时，必须先进行意图探查：

1. **高抽象动词且缺少可验证目标**：优化、改进、提升、完善、处理、搞定。
2. **模糊目标**：让它更好、更快、更稳定、更优雅。
3. **缺少WHY**：只说做什么，不说为什么要做。
4. **隐含假设**：用户预设了解决方案但没说明原始问题。

#### 跳过条件

满足以下任一条件时，可以跳过意图探查，直接进入五维诊断和智能路由：

- 用户请求已经包含目标、约束和验收标准。
- 任务低风险、目标单一，且缺失信息不会改变交付物或执行路径。
- 用户明确使用 `[直出]`。

#### 探查框架

向用户展示一个"意图决策树"，通过 2-3 个问题定位真实意图。

**第1问：目标层**

"你希望这个任务最终解决什么问题？"

- A. 解决当前遇到的具体故障/bug
- B. 提升某个可量化的指标（性能/安全/可读性）
- C. 学习/理解某个概念或最佳实践
- D. 为后续更大的改动做准备
- E. 满足外部要求（代码审查/合规/团队规范）

**第2问：约束层**（根据第1问答案动态生成）

- 如果选 A（故障）→ "这个问题在什么场景下出现？影响范围多大？"
- 如果选 B（指标）→ "当前值是多少？目标值是多少？为什么是这个目标？"
- 如果选 C（学习）→ "你希望最终获得可执行代码，还是带详细解释的方案？"

**第3问：验收层**

"怎样算是真正解决了这个问题？"

这一问必须逼迫用户说出可验证的成功标准。

#### 探查输出

```text
## 意图探查结果

**表面需求**：[用户原话]
**真实意图**：[从决策树提取的根本目标]
**核心约束**：[用户未明说但影响方案的限制]
**验收标准**：[用户确认的成功定义]

**意图确认**：[向用户复述]
"根据你的回答，你真正需要的是【真实意图】，
而不是单纯的【表面需求】。对吗？"

→ 用户确认后，才进入下一阶段
```

## 2. 五维诊断（增强版）

当前问题：评分过于主观，AI 容易给高分。因此五维诊断必须采用零容忍评分标准。

### 严格诊断标准（Zero-Tolerance Scoring）

### D1 精确性（Precision）

- 2分：动词精确（实现/重构/审查）+ 对象具体（文件路径/函数名）+ 目标可量化
- 1分：动词或对象有一项模糊
- 0分：存在"帮我/看看/处理/优化"等高抽象动词

### D2 约束性（Constraint）

- 2分：明确3项以上（范围/数量/权限/禁止项/兼容性）
- 1分：明确1-2项
- 0分：无任何约束说明

### D3 结构性（Structure）

- 2分：指定交付物格式 + 输出结构 + 篇幅限制
- 1分：仅指定格式或结构
- 0分：未说明任何输出要求

### D4 上下文（Context）

- 2分：包含技术栈/当前状态/用途/受众4项
- 1分：包含1-2项
- 0分：无背景信息

### D5 验证性（Verification）

- 2分：有可执行的验收标准（测试命令/量化指标/检查清单）
- 1分：有验收概念但不可执行（"要好用"）
- 0分：无验收标准

**零容忍规则**：

- 总分<6分 → 禁止直接优化，必须先澄清或探查
- 任一维度0分 → 该维度必须补全到至少1分
- D5=0 → 强制补全验收标准，无论用户是否提及

### 诊断输出（增强版）

| 维度 | 得分 | 缺失项 | 强制补全动作 |
| --- | --- | --- | --- |
| D1 精确性 | 0/2 | 动词模糊、对象不明 | → 追问：具体是哪个文件的哪个函数？ |
| D2 约束性 | 0/2 | 无范围、权限、禁止项 | → 补全：允许 breaking change 吗？修改范围？ |
| D3 结构性 | 1/2 | 未指定篇幅限制 | → 补全：输出长度预期（简要/详细/完整） |
| D4 上下文 | 1/2 | 缺少用途和受众 | → 追问：这个结果给谁用？用于什么场景？ |
| D5 验证性 | 0/2 | 无验收标准 | → 强制：怎么判断做对了？测试？指标？ |
| **总分** | **2/10** | **严重不足** | **禁止直接优化，进入澄清访谈** |

**执行决策**：

- 得分<6 且 D5=0 → 澄清访谈模式
- 得分<6 且包含高风险信号 → 意图探查模式
- 得分6-7 → Agent Brief 模式（强制补全所有0分项）
- 得分8-10 → 基础优化模式

## 3. Agent Brief 八组件

面向 agent 的最终输出，应优先组装为以下结构：

| 组件 | 作用 | 问题 |
| --- | --- | --- |
| **INTENT** | 用户真正想达成什么 | 为什么要做这件事？ |
| **CONTEXT** | 背景和已有状态 | agent 需要先知道什么？ |
| **TARGET** | 具体任务对象 | 改什么、写什么、分析什么？ |
| **SCOPE** | 范围和边界 | 包括什么，不包括什么？ |
| **DELIVERABLES** | 交付物 | 最终要输出什么形态？ |
| **CONSTRAINTS** | 技术、风格、权限、反面约束 | 不允许做什么？必须保持什么？ |
| **WORKFLOW** | 执行策略 | 直接执行、先问、先设计、分阶段？ |
| **ACCEPTANCE** | 验收与自检 | 怎么判断做对了？ |

如果任务涉及代码库、长期项目或多轮协作，再追加：

- **MEMORY**：需要沉淀到项目规则、上下文或经验库的内容。
- **RISKS**：最容易误解、最容易做坏的点。

## 4. 澄清优先级

不是所有缺失信息都值得追问。按以下顺序判断：

1. **会改变目标的缺失**：必须问。
2. **会改变架构或交付物的缺失**：必须问。
3. **会影响安全、数据、成本、权限的缺失**：必须问。
4. **只影响表达风格或细节的缺失**：可以给默认假设并继续。
5. **可通过阅读代码、文档、网页获得的信息**：agent 应自己探索，不问用户。

澄清规则：

- 一次只问一个问题。
- 每个问题都给出推荐答案。
- 问题必须能减少执行偏差。
- 不问“你还有什么补充吗”这种低价值问题。

## 4.5 意图-表达偏差（Intent-Expression Gap）

**核心洞察**：用户的自然语言经常存在三类系统性偏差，优化器应主动检测。偏差检测不是否定用户，而是防止把表层表达误当成已确认的真实目标。

### 偏差检测规则

### 偏差A：解决方案伪装成需求（XY Problem）

**信号**：用户直接要求一个具体做法，但做法本身很反常。

- "帮我用正则解析这个HTML" → 真实问题可能是"提取网页数据"，正则是错误路径
- "帮我把这个全局变量改成单例" → 真实问题可能是"状态管理混乱"

**处理**：不直接拒绝用户的方案，而是输出：

```text
我可以按你说的做【X】。不过如果你的最终目的是【推测的Y】，
【Z方案】通常更合适。请确认：A. 就按X做 B. 改用Z做
```

必须把【推测的Y】标为推测或待确认，不得当作用户已经确认的事实。

### 偏差B：症状伪装成原因

**信号**：用户描述的是表象，把表象当成了要修复的对象。

- "这个页面加载慢，帮我加个loading动画" → 真实问题可能是接口性能
- "帮我把这个报错信息隐藏掉" → 真实问题是错误本身

**处理**：在 Agent Brief 的"背景"中同时记录症状和可能的根因，在"工作方式"中要求 agent 先诊断根因再决定修复层次。

### 偏差C：局部视角遮蔽全局目标

**信号**：用户盯着一个细节，但该细节的修改会影响更大范围。

- "把这个函数的参数改成对象" → 会影响所有调用方
- "把这个配置改成环境变量" → 会影响部署流程

**处理**：在 Agent Brief 的"范围"中强制加入影响面分析：

```text
执行前先列出此修改的所有影响点，超出【范围】的影响需要先报告
```

## 4.6 共识快照（Consensus Snapshot）

**核心洞察**：澄清访谈的结论如果不固化，多轮对话后就会漂移。

### 共识快照协议

每完成一轮澄清（无论是意图探查还是单点澄清），输出一个快照块：

```markdown
## 📌 共识快照 #N

**已确认**：
- 真实目标：[用户确认的意图]
- 关键决策：[用户选择的方向]
- 明确排除：[用户说不要的]

**仍待定**：
- [尚未确认的点，标注默认假设]

**基于以上共识继续。如有偏差请随时纠正。**
```

规则：

- 快照编号递增，后续快照只记录增量变化。
- 最终的 Agent Brief 必须与最新快照一致。
- 用户纠正快照 → 立即更新，并检查已有转换是否需要回滚。

## 4.7 置信度标注（Confidence Annotation）

**核心洞察**：优化器补全的信息有不同置信度，应该让用户一眼看出哪些是"你说的"、哪些是"我补的"、哪些是"我猜的"。

### 置信度三级标注

在优化后的 Prompt 中（或改动记录里）标注信息来源：

- **[原文]**：用户明确说过的，逐字或等义保留。
- **[推断]**：从上下文/惯例合理推断的，用户应扫一眼确认。
- **[假设]**：无依据的默认值，用户必须确认，否则 agent 执行时要先问。

示例改动记录：

```markdown
1. [原文] 目标：重构 parseConfig() 函数
2. [推断] 技术栈：JavaScript（从文件扩展名 .js 推断）
3. [假设] 允许修改内部实现但不允许改函数签名 —— **请确认此假设**
```

规则：

- `[假设]` 超过2条 → 不应直接输出优化结果，应转入澄清访谈。
- `[假设]` 不超过2条 → 可以先输出，但必须显式标注待确认，并写入 Agent Brief 的"执行时先验证"指令。
- 所有 `[假设]` 必须在 Agent Brief 中转化为"执行时先验证"的指令。
- `[假设]>2条 且未转澄清 → 输出无效，必须重做` 是硬性门槛，不是建议。

## 阶段4：契约回验（Contract Verification）

**问题**：转换完成≠转换正确。优化后的 Prompt 可能结构完美，但偏离了用户原始意图。

**解决方案**：在输出前增加强制回验环节，作为输出前最后一道门。

### 契约回验协议（输出前最后一道门）

### 回验四问（每次输出前必须自答）

**Q1 意图保真**：优化后的 Prompt 解决的还是用户的原始问题吗？

- 检查方法：把优化后的"目标"部分与意图探查的"真实意图"对照。
- 失败信号：目标被替换成了"更规范但不是用户要的"版本。

**Q2 无擅自决策**：转换过程中是否替用户做了产品/技术决策？

- 检查方法：列出所有补全的信息，逐条判断是"结构补全"还是"方向决策"。
- 结构补全（允许）：补格式、补验收、补执行策略。
- 方向决策（禁止）：选技术方案、定优先级、改目标。

**Q3 可独立执行**：一个对上下文一无所知的 agent，拿到这份契约能不猜测地执行吗？

- 检查方法：模拟新 agent 视角，找出所有需要"猜"的点。
- 失败信号：存在"视情况而定""合理选择"等甩锅表述。

**Q4 验收可判定**：任务完成时，能客观判断做对了还是做错了吗？

- 检查方法：每条验收标准问"这条能用命令/数字/清单检查吗？"
- 失败信号：验收标准里有"良好""合理""优雅"等主观词。

### 回验失败处理

以下是流程动作，不是提醒：

- Q1失败 → 回到意图探查，重新确认。
- Q2失败 → 把擅自决策的点改写为"澄清问题+推荐答案"。
- Q3失败 → 补全模糊点，或在 Prompt 中写明"遇到X时如何处理"。
- Q4失败 → 重写验收标准为可执行形式。

### 回验输出（附在诊断之后）

```markdown
## 契约回验

- ✅ Q1 意图保真：目标与用户确认的真实意图一致
- ✅ Q2 无擅自决策：所有补全均为结构性，技术选型已转为澄清问题
- ✅ Q3 可独立执行：无需猜测的点：0
- ✅ Q4 验收可判定：5条验收标准均可用命令或清单检查
```

## 5. 自主思维循环

复杂任务应要求 agent 使用六阶段循环：

1. **感知**：阅读需求、上下文、相关文件，区分用户原话和真实目标。
2. **自设计**：判断需要什么角色、能力和策略。
3. **深度思考**：比较方案、识别风险、确定最小可行路径。
4. **执行**：小步实现，每步可验证。
5. **反思**：从技术、业务、架构、体验四个维度自检。
6. **沉淀**：记录项目模式、踩坑、规则建议和后续可复用模板。

这不是要求 agent 展示冗长推理，而是要求它采用稳定的工作过程，并把关键决策和验收结果显性化。

## 6. 转换规则

核心原则：信息不足时，宁可停下来澄清，也不要猜测和妥协。转换规则不是润色技巧，而是把原始请求补成可执行契约的强制执行清单。

### 转换规则执行清单（Zero-Compromise Transformation）

执行前必须自检：

- [ ] 意图探查已完成（如触发）。
- [ ] 五维诊断所有维度≥1分。
- [ ] D5验证性≥1分（强制）。
- [ ] 高风险信号已处理（如有）。
- [ ] 用户已确认真实意图（如有歧义）。

未满足以上任一项时，不得输出最终优化结果；必须先澄清、探查或补全到可执行状态。

### R1 精确性注入（强制执行）

触发：检测到模糊动词（优化、改进、处理、看看、弄一下、帮我）。

禁止行为：

- 不得用"可能""也许""建议"等词替代明确目标。
- 不得自行猜测用户意图。
- 不得用"根据常规做法"替代用户确认或可验证上下文。

操作：替换为"精确动词 + 具体对象 + 目标结果"。如果对象或目标缺失且会改变结果，先问一个问题；如果可从文件、文档、日志、URL获得，写入 agent 的感知步骤。

示例：

- 原始：帮我优化这个项目
- 优化：审计当前项目的文档结构和 prompt 转换流程，找出导致 agent 误解用户意图的缺口，并重构核心方法论和模板。

### R2 边界锁定（强制执行）

触发：缺少范围、权限、数量、排除项。

禁止行为：

- 不得用"尽量不改"替代"禁止修改"。
- 不得用"一般来说"替代明确权限。
- 不得假设用户"应该允许"某些修改。

操作：补充"只做什么 / 不做什么 / 需要确认后才能做什么"。涉及文件修改、联网、删除、提交、部署、外部系统写入时，必须写清权限边界。

### R3 交付物锁定（强制执行）

触发：未指定输出格式。

禁止行为：

- 不得用"灵活输出"替代明确格式。
- 不得让 AI 自行决定详细程度。

操作：指定最终交付形态、结构和篇幅，如 Agent Brief、PRD、代码补丁、审查报告、决策矩阵、JSON。详细程度必须以"简要/完整/逐项/限定字数"等可执行方式表达。

### R4 上下文锚定（强制执行）

触发：缺少背景、受众、用途、项目状态。

禁止行为：

- 不得假设"用户肯定知道自己的技术栈"。
- 不得省略"显而易见"的背景。

操作：补充最小必要背景，包括用户场景、当前状态、受众、用途。可由 agent 自行读取的上下文必须写成"先读取 X"，不可转嫁给用户。

### R5 反面约束（强制执行）

触发：无禁止项说明，或 agent 自由度过大。

禁止行为：

- 不得用"注意"替代"禁止"。
- 不得用"建议"替代"必须"。

操作：添加 1-3 条最关键的"不要做 / 禁止 / 必须保持"。优先限制最容易造成损失的行为，并从 `templates/ANTI-PATTERNS-REFERENCE.md` 选择任务类型匹配的反面约束。

### R6 范围裁剪（强制执行）

触发：一个指令包含3个以上独立任务，或任务之间目标、交付物、验收方式明显不同。

禁止行为：

- 不得试图"一次性完成所有任务"。
- 不得擅自决定哪些任务可以合并。

操作：拆成主任务、次任务、后续任务。当前只让 agent 处理一个主目标；合并或改优先级会改变结果时，先问一个问题并给推荐答案。

### R7 示例嵌入（条件执行）

触发：主观任务、风格任务、复杂格式任务没有例子。

禁止行为：

- 不得用"参考类似风格"替代具体示例。

操作：给 1 个正例，必要时给 1 个反例。示例必须服务验收和边界，不得引入新的目标或替用户决定方向。

### R8 验证门（强制执行，最高优先级）

触发：D5=0 或无明确验收标准。

禁止行为：

- 不得使用抽象验收标准（"要好用""要稳定"）。
- 不得省略验收标准（即使用户未提及）。
- 不得使用不可执行的标准（"符合最佳实践"）。

操作：按任务类型从 `templates/ACCEPTANCE-CHECKLIST.md` 选择可执行验收项，添加可检查的验收项、自检项、测试命令、量化指标或人工检查清单。

最小清单：

- 代码类：测试命令、类型检查、代码质量、新依赖说明、public API 变更标记。
- 调研类：覆盖维度、来源依据、明确推荐、场景匹配。
- 写作类：篇幅、语气、结构、术语准确性。
- 重构类：现有测试、兼容性、复杂度或重复改善、修改范围和风险说明。
- 部署类：部署前检查、环境配置、回滚条件、监控日志、版本和影响范围。

### R9 意图路由（强制执行）

触发：所有任务。

必须写入每个 Agent Brief：

```markdown
## 执行策略

- 信息足够且低风险 → 直接执行。
- 关键信息缺失且会改变目标 → 停下来，只问一个问题，附推荐答案。
- 信息可从文件/文档/日志/URL获得 → 自己读取，不问用户。
- 高风险操作（删除/生产/不可逆）→ 先输出方案和影响面，等待确认。
- 只读分析任务 → 禁止修改任何文件。
- 代码修改任务 → 先读上下文，再最小变更。
```

禁止行为：

- 不得省略"信息不足时怎么办"的指引。
- 不得让 agent 自行决定是否需要确认高风险操作。

### R10 沉淀门（条件执行）

触发：长期项目、代码库任务、团队规则、重复性任务。

禁止行为：

- 不得输出"本次任务很顺利"式的空洞沉淀。
- 每条沉淀必须能改变下次 agent 的行为，否则删掉。

操作：要求 agent 在完成后输出"应沉淀的项目规则 / 术语 / 风险 / 模板"。沉淀项必须包含适用场景、触发条件和下次执行规则。

## 7. 智能路由

用户不应该记忆模式名。优化器应先自动判断任务类型，再选择输出形态。

## 阶段2：智能路由（重构版）

**当前问题**：路由规则模糊，AI容易误判。

**解决方案**：决策树式路由 + 强制触发器。

### 智能路由决策树 v2.0

#### 第一层：安全阀判断（强制优先级）

```text
用户指令
  │
  ├─ 包含高风险信号？
  │  ├─ 是，且缺少目标、范围、约束或验收 → 强制进入【意图探查】
  │  │  信号：生产环境/数据库/删除/重构（目标或范围不明时）/越X越好/提升性能（指标不明时）
  │  ├─ 是，但目标、范围、约束和验收已明确 → 继续路由，并在 Agent Brief 中加入影响面分析和风险确认
  │  └─ 否 → 继续
  │
  ├─ 诊断总分<6？
  │  ├─ 是 → 强制进入【澄清访谈】
  │  └─ 否 → 继续
  │
  └─ D5验证性=0？
     ├─ 是 → 强制补全验收标准
     └─ 否 → 继续
```

#### 第二层：任务类型判断

```text
通过安全阀
  │
  ├─ 编程类任务？
  │  关键词：实现/重构/调试/部署/测试
  │  → 【Agent Brief - 代码模板】
  │
  ├─ 调研类任务？
  │  关键词：对比/分析/调研/评估/选型
  │  → 【Agent Brief - 调研模板】
  │
  ├─ 写作类任务？
  │  关键词：写/撰写/文档/PRD/报告
  │  → 【Agent Brief - 写作模板】
  │
  ├─ 项目沉淀？
  │  关键词：记住/下次/以后/规则/团队约定
  │  → 【项目上下文模板】
  │
  └─ 其他
     → 【通用Agent Brief】
```

#### 第三层：复杂度判断

```text
确定任务类型
  │
  ├─ 单步简单任务？
  │  特征：改一个文件/单一动作/无依赖
  │  → 【基础优化模式】
  │
  ├─ 多步复杂任务？
  │  特征：3+文件/跨模块/需要架构设计
  │  → 【Agent Brief + 分阶段审批门】
  │
  └─ 长期迭代任务？
     特征：会有后续版本/需要上下文沉淀
     → 【Agent Brief + 项目上下文沉淀】
```

### 输出模式映射

| 模式 | 触发条件 | 输出 |
| --- | --- | --- |
| Basic Optimize / 基础优化 | 通过安全阀，且第三层判定为单步简单任务 | 优化后的 Prompt + 简短诊断 + 路由决策日志 + 改动记录 |
| Agent Brief | 开发、重构、调研、设计、调试、执行类多步骤任务 | 完整任务简报，包含目标、背景、范围、交付物、工作方式、约束、验收标准和沉淀要求 |
| Clarify First / 先澄清 | 安全阀命中低分、关键缺失或 `[假设]` 超过2条 | 当前理解、最大不确定点、一个问题、推荐答案、共识快照 |
| Project Context / 项目上下文 | 用户想沉淀规则、术语、团队约定、项目记忆 | 可保存的项目上下文 / 记忆文件草案 |
| Direct Output / 直出 | 用户明确 `[直出]`、只给结果，且未命中安全阀 | 只输出优化后的 Prompt |

### 路由日志（必须输出）

除非用户使用 `[直出]` 且未命中安全阀，否则必须输出路由决策日志：

```markdown
## 路由决策日志

**第一层：安全阀** ✅ 通过
- 高风险信号：未检测到
- 诊断总分：8/10（≥6）
- 验证性：2/2（已明确）

**第二层：任务类型** → 编程类任务
- 检测关键词："重构"
- 匹配模板：Agent Brief - 代码模板

**第三层：复杂度** → 单步简单任务
- 影响范围：单文件
- 依赖关系：无跨模块依赖
- 最终模式：基础优化模式

**执行路径**：基础优化模式 + 代码验收清单
```

路由原则：

- 不让用户选择模式，除非用户主动指定。
- 简单任务走基础优化，不输出过重结构。
- 多步骤 agent 任务走 Agent Brief。
- 高风险不确定任务先澄清，一次只问一个问题。
- 长期项目和重复协作任务自动加入沉淀门。
- 能通过读取代码、文档、日志、网页获得的信息，不问用户，写入未来 agent 的感知步骤。

## 8. 严格性保障 / 防退化机制

优化器本身是 AI 执行的，AI 天然倾向于"顺着用户说、快速给结果"。以下机制防止优化器自己退化：

### 1. 禁止性规则前置

所有"禁止行为"条目在 System Prompt 中必须出现在对应"执行规则"之前。靠前的约束更容易被遵守，不能把禁止项放到附录或弱化为提醒。

### 2. 强制输出证据

- 诊断表不允许出现无"问题"描述的低分，打0分必须说出缺了什么。
- 改动记录每条必须引用规则编号（R1-R10）并说明降低了哪类偏差。
- 路由决策必须输出日志，说明走了哪条分支、为什么。

### 3. 硬性门槛不可协商

以下是重做条件，不是注意事项：

```text
D5=0 且未补全验收 → 输出无效，必须重做
[假设]>2条 且未转澄清 → 输出无效，必须重做（见 4.7 置信度标注）
总分<6 且直接输出了优化结果 → 输出无效，必须重做
高风险信号 且未探查意图 → 输出无效，必须重做
```

### 4. 反奉承约束

- 用户的原始指令得分低不是冒犯，如实打分。
- 不要为了显得高效而跳过澄清。一次正确的澄清比三轮返工便宜。
- 用户坚持一个可能错误的方案时，执行前必须留一句风险记录在 Agent Brief 中。

## 9. 反模式

### 1. 只润色，不对齐

把“帮我做个功能”改成“请实现一个功能”，仍然没有解决目标、边界、验收缺失。

### 2. 替用户做产品决策

优化 prompt 可以补全结构，但不能擅自改变用户目标。

### 3. 一次问太多

多个问题会让用户难以回答，也会隐藏最关键的决策点。

### 4. 默认直接执行

高风险任务必须先对齐方案和验收。

### 5. 过度规则化

prompt 太长、约束太多，会让 agent 执行迟钝。只保留能改变行为的规则。

### 6. 缺少验证门

没有验收标准的任务，agent 很容易提前宣布完成。

### 7. 缺少上下文记忆

每次都重新解释项目，会让 agent 重复犯错。长期项目必须沉淀术语、架构和决策。

---

## 10. 最终自检清单

发送给 agent 前，检查：

- [ ] agent 是否知道真正目标，而不只是表面任务？
- [ ] 是否说明了必须读取或参考的上下文？
- [ ] 是否明确了范围、边界和禁止事项？
- [ ] 是否定义了交付物格式？
- [ ] 是否设置了信息不足时的行为？
- [ ] 是否有可检查的验收标准？
- [ ] 是否要求复杂任务完成后反思和沉淀？
- [ ] Q1 意图保真：优化后的 Prompt 解决的还是用户的原始问题吗？
- [ ] Q2 无擅自决策：转换过程中是否替用户做了产品/技术决策？
- [ ] Q3 可独立执行：一个对上下文一无所知的 agent，拿到这份契约能不猜测地执行吗？
- [ ] Q4 验收可判定：任务完成时，能客观判断做对了还是做错了吗？

如果 3 项以上无法打勾，先优化或进入澄清访谈。
